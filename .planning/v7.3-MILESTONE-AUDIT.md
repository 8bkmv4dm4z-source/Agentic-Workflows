## Tech Debt by Phase

### Phase 07.3: Semantic Context System

#### TD-1: ArtifactStore is Dead Code at Runtime (SCS-07)

**File:** `src/agentic_workflows/storage/artifact_store.py`

`ArtifactStore` was built (Wave 2B, Plan 04) and the unit test passes. But neither `run.py`, `user_run.py`, nor `graph.py` instantiate it. No `artifact_store` parameter exists on `LangGraphOrchestrator.__init__` or `ContextManager.__init__`. The `MissionContext.artifacts` list (populated at context_manager.py:492) is computed but never persisted anywhere.

To wire: add `artifact_store: ArtifactStore | None = None` to `LangGraphOrchestrator` and `ContextManager`, then call `artifact_store.upsert(run_id, mission_id, key, value, embedding)` inside `_persist_mission_context()` for each artifact.

#### TD-2: Synchronous Postgres Blocking in Planner Hot Path (SCS-09, SCS-10)

**File:** `src/agentic_workflows/orchestration/langgraph/context_manager.py:597`

`query_cascade()` is a synchronous psycopg3 call inside `build_planner_context_injection()`, which is called from the planner node (`graph.py:832`) on every planning step. The hard-timeout in `_generate_with_hard_timeout()` protects only the LLM call — the preceding `query_cascade()` call has no timeout guard.

A slow or stalled Postgres connection (pool exhausted, network stall) will block the planner node indefinitely. The `except Exception` at the bottom of `build_planner_context_injection()` catches errors but not a query that returns slowly.

To mitigate: wrap `query_cascade()` call in a `concurrent.futures.ThreadPoolExecutor` with a short timeout (e.g., 2s), falling back to an empty list on timeout. Alternatively, make `query_cascade()` accept a `statement_timeout` hint and pass it as a Postgres session parameter.

#### TD-3: Cache Memory Leak in Long-Lived Processes (SCS-12)

**File:** `src/agentic_workflows/orchestration/langgraph/context_manager.py:249`

`self._cascade_cache` (and `self._embed_cache`) are instance-level dicts that grow unboundedly. Cache key format: `f"{run_id}:{goal_text}"`. In `UserSession` (which reuses the same `LangGraphOrchestrator` instance across calls) and the fork-test batch runner (`run.py:_run_fork_test_batch`), each `orchestrator.run()` call with a unique `run_id` adds new, never-evicted entries.

For a 10-run fork-test with 5 missions and 20 planner steps: up to 1,000 unique `run_id:goal` keys accumulate per session.

To fix: implement LRU eviction (e.g., `cachetools.LRUCache(maxsize=100)`) or clear the cache at the start of each `run()` call (simpler: `self._cascade_cache.clear()` at the top of `build_planner_context_injection()` when `current_step == 1`).

#### TD-4: Missing `source_layer` in Unit Test Fixture (Minor)

**File:** `tests/unit/test_context_manager_7_3.py:15-24`

`_make_result()` constructs a `MissionContextResult` dict without the required `source_layer` field. All unit test assertions check for `[Cross-run] Similar:` prefix only, so the missing field doesn't cause failures — but attribution always shows `'?'` in unit test runs. Only the Postgres integration tests (which go through the real `_row_to_result()` path) validate attribution correctly.

To fix: add `"source_layer": "L0"` (or `"mock"`) to `_make_result()` to match the TypedDict contract.
