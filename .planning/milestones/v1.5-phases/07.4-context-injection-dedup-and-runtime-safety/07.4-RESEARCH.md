# Phase 07.4: Context Injection Dedup and Runtime Safety - Research

**Researched:** 2026-03-08
**Domain:** Python concurrency, LRU-bounded caches, TypedDict fixture hygiene, ContextManager message dedup
**Confidence:** HIGH

## Summary

Phase 07.4 is a focused correctness and safety hardening phase on the `ContextManager` class
(`src/agentic_workflows/orchestration/langgraph/context_manager.py`). It addresses four
independent bugs introduced or exposed by Phase 07.3 cross-run injection work.

The most disruptive bug is the cross-run dedup failure: `build_planner_context_injection()`
appends a `[Cross-run] Similar:` message to `state["messages"]` on **every planner step** because
there is no tracking of what has already been injected. After N planner steps, N copies of the
same cross-run text accumulate in the message history, consuming context window budget and
potentially confusing the LLM. The fix requires a new per-run instance set
(`_injected_cross_run_keys`) that gates injection at the message-append call site in `graph.py`.

The second bug is a latency risk: `query_cascade()` is called synchronously in the planner hot
path with no timeout guard. A slow or hung Postgres connection blocks the entire planner step
indefinitely. The fix wraps the call in `concurrent.futures.ThreadPoolExecutor` with a 2-second
`Future.result(timeout=2)` fallback — the same pattern the existing
`_generate_with_hard_timeout()` uses (with `threading.Thread` + `queue.Queue`), but using the
higher-level futures API.

The third bug is a memory leak: `_cascade_cache` and `_embed_cache` are plain `dict` instances
that grow without bound in long-lived processes (e.g., the FastAPI service running for hours with
many distinct `run_id:goal` combinations). Both caches must be capped at 200 entries with
truncation on overflow (delete oldest half when cap is hit).

The fourth bug is a test fixture omission: `_make_result()` in
`tests/unit/test_context_manager_7_3.py` constructs a `MissionContextResult` TypedDict without
the required `source_layer` field. `MissionContextResult` has `source_layer: str` as a required
key. Its absence causes the `attribution_parts` log in `build_planner_context_injection()` to
emit `?` for the layer label in every unit test, masking real attribution data.

**Primary recommendation:** Implement all four plans sequentially. Each plan touches exactly one
logical concern and can be tested in isolation with a single targeted test class.

### Pattern 1: Per-run Injection Key Guard (Plan 01)

**What:** Add `_injected_cross_run_keys: set[str]` as an instance attribute on `ContextManager`.
Before appending a cross-run injection to `state["messages"]` in `graph.py`, check whether the
`cache_key` (already computed as `f"{run_id}:{goal_text}"`) is in this set. If present, skip the
append. If not present, append and add to the set.

**When to use:** Every call to `build_planner_context_injection()` that returns a non-empty
cross-run result.

**Where the guard lives:** The set tracks *injection into messages*. The message append happens
in `graph.py` line 834:
```python
context_injection = self.context_manager.build_planner_context_injection(state)
if context_injection:
    state["messages"].append({"role": "user", "content": context_injection})
```
Two implementation options:
1. Guard inside `build_planner_context_injection()` — returns empty string on repeated call for
   same key, so `graph.py` code is unchanged.
2. Return the cross-run text each time but add a guard field; `graph.py` checks it.

Option 1 is cleaner: the method returns `""` for the cross-run portion when `cache_key` is already
in `_injected_cross_run_keys`, so no change needed in `graph.py`. The `_injected_cross_run_keys`
set is added to `__init__`.

**Key detail:** The existing `_cascade_cache` already deduplicates the SQL call. This new set
deduplicates the *message append*. They are separate concerns. The cascade cache prevents N SQL
calls; the injection key set prevents N message appends. Both are needed.

**Key detail:** `_injected_cross_run_keys` is an **instance-level** set, not stored in
`state["policy_flags"]`. The MEMORY.md note about `injected_mission_ids` refers to prior-mission
summary dedup (a different mechanism); cross-run injection dedup is new and lives on the
ContextManager instance.

### Pattern 2: ThreadPoolExecutor Timeout for query_cascade() (Plan 02)

**What:** Wrap the `self._store.query_cascade(...)` call inside `build_planner_context_injection()`
with a `concurrent.futures.ThreadPoolExecutor` submit + `Future.result(timeout=2.0)`. On timeout,
log a warning and fall back to `[]`.

**When to use:** Whenever `self._store is not None` and cascade is about to be called.

**Canonical pattern:**
```python
from concurrent.futures import ThreadPoolExecutor, TimeoutError as FuturesTimeoutError

_CASCADE_TIMEOUT_SECONDS = 2.0

with ThreadPoolExecutor(max_workers=1) as executor:
    future = executor.submit(
        self._store.query_cascade, goal_text, embedding=embedding, top_k=3
    )
    try:
        hits = future.result(timeout=_CASCADE_TIMEOUT_SECONDS)
    except (FuturesTimeoutError, Exception):
        self._logger.warning("query_cascade timeout/error — falling back to []")
        hits = []
```

The `ThreadPoolExecutor` context manager ensures thread cleanup. The existing exception-catching
in `build_planner_context_injection()` already wraps the cascade block with `except Exception` —
the timeout wrapper fits inside that block.

**Important:** `concurrent.futures.TimeoutError` is `builtins.TimeoutError` in Python 3.11+ but
should be imported as `from concurrent.futures import TimeoutError as FuturesTimeoutError` for
clarity and forward-compat. In Python 3.12 (project's runtime), both are the same object.

### Pattern 3: Bounded Cache with Overflow Truncation (Plan 03)

**What:** When `len(self._cascade_cache) >= _CACHE_MAX_SIZE`, truncate by deleting the oldest
half before inserting a new entry. Same for `_embed_cache`.

**Cap constant:** `_CACHE_MAX_SIZE: int = 200` at module level (follows the project's pattern of
module-level cap constants from Phase 7.2, e.g., `_PIPELINE_TRACE_CAP`).

**Truncation on overflow:**
```python
_CACHE_MAX_SIZE: int = 200

# Before inserting into _cascade_cache:
if len(self._cascade_cache) >= _CACHE_MAX_SIZE:
    # Delete oldest half (dict preserves insertion order in Python 3.7+)
    remove_count = _CACHE_MAX_SIZE // 2
    for key in list(self._cascade_cache.keys())[:remove_count]:
        del self._cascade_cache[key]
```

Python `dict` preserves insertion order (guaranteed since Python 3.7), so the oldest keys are
first. This is a simple FIFO eviction, not LRU, but it is sufficient for a bounded cache with
infrequent key diversity.

The same pattern applies to `_embed_cache`.

**Test:** Instantiate `ContextManager`, call `build_planner_context_injection` 300 times with
distinct `run_id:goal` pairs (using a MagicMock store that returns `[]`). Assert
`len(cm._cascade_cache) <= _CACHE_MAX_SIZE` and `len(cm._embed_cache) <= _CACHE_MAX_SIZE`.

### Pattern 4: Fix _make_result() Missing source_layer (Plan 04)

**What:** `MissionContextResult` is a TypedDict with 8 required keys (see `mission_context_store.py`
lines 127-138):
```
id, run_id, mission_id, goal, summary, tools_used, score, source_layer
```

Current `_make_result()` in `test_context_manager_7_3.py` constructs only 7 keys — omitting
`source_layer`. TypedDict does not enforce missing keys at runtime (Python TypedDicts are
structural, not enforced at construction), so the omission is silently accepted. But the
`attribution_parts` computation in `build_planner_context_injection()` does:
```python
f"{hit.get('source_layer', '?')}:{hit.get('score', 0.0):.2f}"
```
Because `source_layer` is missing, `.get()` returns `'?'`, so attribution logs always show `?:0.90`
in unit tests even when a real layer should appear.

**Fix:** Add `source_layer="L0"` (or any valid layer label) to `_make_result()`:
```python
def _make_result(goal: str, summary: str) -> MissionContextResult:
    return MissionContextResult(
        id=1,
        run_id="prev-run",
        mission_id="m-1",
        goal=goal,
        summary=summary,
        tools_used=["write_file"],
        score=0.9,
        source_layer="L0",   # ADD THIS
    )
```

Then add an assertion in `TestCrossRunInjection` that verifies the attribution log line contains
a real layer label, not `?`.

### Anti-Patterns to Avoid

- **Storing `_injected_cross_run_keys` in `state["policy_flags"]`:** State is serialized by the
  LangGraph checkpointer; a `set[str]` is not JSON-serializable. Use instance attribute only.
- **Using `lru_cache` on `build_planner_context_injection`:** It takes `state` (a mutable dict)
  as argument; `lru_cache` requires hashable args.
- **Raising on `FuturesTimeoutError`:** Must silently fall back to `[]` — cascade timeout must
  never propagate to the planner. The existing outer `except Exception` block provides a second
  safety net.
- **Creating a new `ThreadPoolExecutor` per call without cleanup:** Always use as a context
  manager (`with ThreadPoolExecutor(...) as executor:`) to ensure thread cleanup. If performance
  becomes a concern, a module-level shared executor is fine, but the context-manager pattern is
  simpler and correct for the 2s timeout use case.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Thread timeout for sync DB call | Custom `threading.Event` / `multiprocessing.Pool` | `concurrent.futures.ThreadPoolExecutor` + `Future.result(timeout=N)` | Stdlib, clean API, exception propagation handled |
| Bounded cache | `OrderedDict` + LRU logic | Plain `dict` truncation on overflow | Dict preserves insertion order since 3.7; sufficient for FIFO eviction at 200-entry scale |

## Common Pitfalls

### Pitfall 1: Dedup Set Scope Too Broad
**What goes wrong:** If `_injected_cross_run_keys` is a class variable instead of instance
variable, it is shared across all `ContextManager` instances, causing cross-test interference.
**Why it happens:** Python class variables are shared across instances.
**How to avoid:** Initialize in `__init__` as `self._injected_cross_run_keys: set[str] = set()`.
**Warning signs:** Tests that run in isolation pass but fail when run together.

### Pitfall 2: Cascade Cache Key Mismatch
**What goes wrong:** The injection key set uses a different key format from `_cascade_cache`,
so the same goal gets injected multiple times despite being cached at the SQL level.
**Why it happens:** Using `goal_text` alone instead of `f"{run_id}:{goal_text}"`.
**How to avoid:** Use exactly `cache_key = f"{run_id}:{goal_text}"` in both the cascade cache
and the injection key set. This is already the format computed at line 580 of context_manager.py.

### Pitfall 3: FuturesTimeoutError vs TimeoutError Confusion
**What goes wrong:** Python 3.11+ aliases `concurrent.futures.TimeoutError` to `builtins.TimeoutError`.
Catching only one may miss the other in some Python versions.
**Why it happens:** Version-dependent aliasing.
**How to avoid:** `except (TimeoutError, Exception)` catches both; or import
`from concurrent.futures import TimeoutError as FuturesTimeoutError` for explicit clarity.
The existing outer `except Exception` in `build_planner_context_injection()` provides a fallback.

### Pitfall 4: Cache Truncation Mutating Dict During Iteration
**What goes wrong:** `for key in self._cascade_cache: del self._cascade_cache[key]` raises
`RuntimeError: dictionary changed size during iteration`.
**Why it happens:** Python dicts cannot be mutated during iteration.
**How to avoid:** Snapshot the keys first: `for key in list(self._cascade_cache.keys())[:N]: del ...`

### Pitfall 5: TypedDict Missing Key Not Caught at Construction
**What goes wrong:** Adding `source_layer` to `_make_result()` doesn't change runtime behavior
without adding an assertion that checks attribution output.
**Why it happens:** TypedDicts are structural type hints, not enforced at runtime.
**How to avoid:** Add an assertion in the test that verifies the attribution log format directly
(e.g., check that `build_planner_context_injection` return value or attribution log does not
contain `?` when `source_layer` is set).

### Plan 03: Bounded Cache with FIFO Truncation
```python
# Source: module-level constant, context_manager.py

_CACHE_MAX_SIZE: int = 200

def _cache_put(self, cache: dict, key: str, value: object) -> None:
    """Insert into cache, truncating oldest half on overflow."""
    if len(cache) >= _CACHE_MAX_SIZE:
        remove_count = _CACHE_MAX_SIZE // 2
        for k in list(cache.keys())[:remove_count]:
            del cache[k]
    cache[key] = value
```

## Open Questions

1. **Should `_injected_cross_run_keys` be cleared between missions within a run?**
   - What we know: A run has N missions; the cross-run injection fires for the active mission's
     goal. If mission 2 has the same goal as mission 1, the injection would be suppressed for
     mission 2 if the same key is used.
   - What's unclear: Whether same-goal across missions within a run is a realistic scenario.
   - Recommendation: Key includes `run_id` + `goal_text`; same goal same run = same cache_key =
     one injection. This is correct behavior — same goal should not re-inject.

2. **Should the `ThreadPoolExecutor` be a module-level singleton vs. per-call context manager?**
   - What we know: Per-call `with ThreadPoolExecutor(max_workers=1)` creates and destroys a
     thread pool per planner step (but only when cache misses). After the cache is populated,
     no new executor is created.
   - Recommendation: Per-call context manager is simpler and correct; for the 2s timeout use
     case, thread pool creation overhead is negligible.
