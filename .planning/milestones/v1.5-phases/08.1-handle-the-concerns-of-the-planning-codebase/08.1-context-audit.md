# ContextManager Architecture Audit

**Phase:** 08.1 ŌĆö Handle the Concerns of the Planning Codebase
**Authored:** 2026-03-12T00:00:00Z
**Status:** specification (no code changes in Phase 8.1)
**Unblocks:** Phase 8.2 ŌĆö ContextManager Redesign

## Overview

The `ContextManager` class (`src/agentic_workflows/orchestration/langgraph/context_manager.py`, ~1054 lines)
is responsible for maintaining mission context across the lifetime of a multi-mission agent run.
It tracks per-mission tool results, compresses context on mission completion (evicting old messages),
and injects summarized context into the planner's message window when a new mission begins.

An audit is needed because users report persistent **sub-task amnesia**: after the agent completes
several sub-phases successfully, it loses awareness of which sub-task produced which result,
causing it to re-issue already-completed tool calls and fail mission completion checks.
The root causes are structural ŌĆö not incidental bugs ŌĆö and require a redesign pass in Phase 8.2.

## Current Architecture

### Core Model: `MissionContext`

Defined in `context_manager.py`. Key fields:

| Field | Type | Purpose |
|-------|------|---------|
| `mission_id` | str | Unique mission identifier |
| `goal` | str | Mission objective text |
| `status` | str | `"active"` \| `"complete"` \| `"partial"` |
| `tools_used` | list[str] | Tool names called during this mission |
| `key_results` | dict[str, str] | Flat dict: tool name ŌåÆ result summary |
| `artifacts` | list[ArtifactRecord] | Structured artifacts extracted from results |
| `sub_missions` | dict[str, SubMissionContext] | Sub-task model objects (rarely populated) |
| `summary` | str | Compressed natural-language summary |
| `step_range` | tuple[int, int] | Graph steps this mission occupied |

### `SubMissionContext` model

Exists on `MissionContext.sub_missions` (dict keyed by sub-task ID). Fields include sub-task
`goal`, `status`, `tools_used`. However, this model is **never populated** by the current
`on_tool_result()` implementation ŌĆö it exists in the schema but not in the lifecycle.

### Key `ContextManager` Methods

| Method | Location | Responsibility |
|--------|----------|---------------|
| `on_tool_result(mission_id, tool, result)` | context_manager.py | Records tool call outcome into `key_results` |
| `on_mission_complete(mission_id, state)` | context_manager.py | Evicts messages, writes compressed summary, persists to store |
| `build_planner_context_injection(state)` | context_manager.py | Returns injection string for next planner message |
| `build_specialist_context(mission_id, state)` | context_manager.py | Returns prior_results_summary for specialist node |
| `_persist_mission_context(ctx)` | context_manager.py | Upserts to `MissionContextStore` and `ArtifactStore` |
| `persist_partial_missions(state)` | context_manager.py | Persists in-progress missions on timeout/finalize |

### State Mutation Touch Points

State mutations to `RunState["mission_contexts"]` are spread across three files:

1. **`context_manager.py`** ŌĆö `on_mission_complete()` writes the completed `MissionContext` dict
   into `state["mission_contexts"]`; `build_specialist_context()` reads from it.

2. **`planner_node.py`** ŌĆö calls `on_tool_result()`, `on_mission_complete()`, and triggers
   auto-inject at `step == 1`. Also calls `mission_tracker.refresh_mission_status()` in the
   same execution block, producing interleaved state writes.

3. **`mission_tracker.py`** ŌĆö `refresh_mission_status()` reads sub-task satisfaction flags from
   `state["missions"]` and writes back updated status. `SUBTASK TICK` log is emitted here.
   This is logically part of context management but architecturally separate.

## Failure Mode 1: Sub-Task Amnesia

### Current behavior

After `on_mission_complete()` evicts the active message window and replaces it with a condensed
summary, the planner's next step sees only the compressed `MissionContext.build_summary()` output.
This summary enumerates tool names and result snippets but does not encode **which sub-task
requested each tool call**. On resume or in a subsequent mission, the planner cannot associate
"sort_array result [1, 2, 3]" with "sub-task 1.2: sort the dataset."

### Root cause

`on_tool_result(mission_id, tool, result)` accepts no `sub_task_id` parameter. It writes the
result into `key_results[tool]` ŌĆö a flat dict keyed only by tool name. The `sub_missions` dict
on `MissionContext` is never populated during the tool-result lifecycle. The summary generation
in `build_summary()` iterates `key_results` items without sub-task attribution.

### Impact

- Agent re-issues previously completed sub-tasks after eviction (duplicate work, wasted tokens).
- Mission completion check (`mission_tracker.refresh_mission_status()`) misses already-satisfied
  sub-tasks because the sub-taskŌĆōresult link was never recorded.
- Multi-mission workloads with overlapping tool usage (e.g., two sub-tasks both call `sort_array`)
  show confusing summaries with only one result entry.

### Proposed fix (Phase 8.2)

Extend `on_tool_result()` signature to accept an optional `sub_task_id: str | None = None`.
When `sub_task_id` is provided, store the result under a new `sub_task_results: dict[str, list[str]]`
field on `MissionContext` keyed by sub-task ID. Update `build_summary()` to emit sub-taskŌĆōattributed
lines: `"[sub-task 1.2] sort_array ŌåÆ [1, 2, 3]"`. The active sub-task ID can be resolved from
`mission_tracker` at the call site in `planner_node.py`.

## Failure Mode 2: Result-to-Subtask Affiliation Gap

### Current behavior

`tool_results` entries in `MissionReport` store `{"tool": ..., "result": ...}` with no sub-task
attribution. `MissionContext.key_results` is a `dict[str, str]` keyed by tool name. When two
different sub-tasks invoke the same tool (e.g., both call `data_analysis`), only the **last
result is retained** in `key_results` (last-write-wins). The first sub-task's result is silently
discarded.

### Root cause

The dict key for `key_results` is the tool name string, not a composite of sub-task ID + tool
name. The model schema has no enforcement that prevents key collision. `on_tool_result()` does
a plain dict assignment: `ctx.key_results[tool] = summarize(result)`.

### Impact

- Auditor's `chain_integrity` check may flag missing results for sub-tasks that were actually
  completed (their result was overwritten).
- `extract_artifacts()` operates on `key_results` ŌĆö artifacts from the first sub-task call are
  permanently lost.
- Re-running the agent on a mission where two sub-tasks share a tool produces non-deterministic
  behavior depending on execution order.

### Proposed fix (Phase 8.2)

Replace `key_results: dict[str, str]` with `sub_task_results: dict[str, dict[str, str]]` where
the outer key is `sub_task_id` and the inner key is tool name. For callers that do not supply
a sub-task ID (backward compat), use a sentinel key `"_unaffiliated"`. Update `extract_artifacts()`
and `build_summary()` to iterate the nested structure. The `mission_reports` `tool_results` list
should also be extended to include `sub_task_id` when available.

## Failure Mode 3: Injection Policy Opacity

### Current behavior

`build_planner_context_injection()` maintains an instance-level `injected_mission_ids: set[str]`
for deduplication ŌĆö once a mission's summary has been injected, it is not re-injected on
subsequent calls. This set is the sole record of what context is currently visible to the planner.
There is no public API to query the injection state. The only way to verify what was injected is
to parse `state["messages"]` looking for `[Orchestrator]`-prefixed entries.

### Root cause

Injection state is implicit ŌĆö encoded as set membership with no observable surface. The design
prioritized simplicity (avoid re-injection bugs) but left the system opaque to tests and operators.
The Phase 07.4 re-injection bug (fixed) was caused by this same opacity: the set was being reset
unintentionally on `on_mission_complete()`, causing the same summaries to be re-injected every step.

### Impact

- Integration tests must parse `state["messages"]` string content to verify injection behavior ŌĆö
  brittle and implementation-coupled.
- Debugging context injection in production requires adding ad-hoc logging.
- The opaque state makes it impossible to implement adaptive injection (e.g., "inject only if
  the planner hasn't seen this mission in the last N steps") without restructuring.

### Proposed fix (Phase 8.2)

Add a `get_injection_state() -> InjectionState` method to `ContextManager` that returns a
lightweight dataclass:

```python
@dataclass
class InjectionState:
    injected_mission_ids: frozenset[str]
    last_injection_chars: int
    cross_run_keys: frozenset[str]
    injection_count: int
```

No behavior change ŌĆö purely an observable read path. Log the full `InjectionState` on each
injection decision at DEBUG level. Tests can assert against `get_injection_state()` without
parsing message content.

## Failure Mode 4: Scattered State Mutation

### Current behavior

`RunState["mission_contexts"]` (a `dict[str, dict]`) is read and written from three different
call sites:

1. `context_manager.on_mission_complete()` ŌĆö writes the completed mission context dict.
2. `context_manager.build_specialist_context()` ŌĆö reads to build specialist context string.
3. `mission_tracker.refresh_mission_status()` ŌĆö reads mission status flags and writes updated
   sub-task satisfaction state back to `state["missions"]` (different key, but logically coupled).
4. `graph.py._plan_next_action()` ŌĆö reads `mission_contexts` for completion detection.

`planner_node.py` orchestrates calls to both `context_manager` methods and `mission_tracker`
in a single execution block, creating an implicit ordering dependency that is not enforced.

### Root cause

No single owner for mission context state. `mission_tracker.py` handles sub-task satisfaction
which is conceptually part of the context lifecycle, but it was implemented as a separate module
to avoid circular imports. The result is that mutation responsibility is split across the codebase
without a documented contract.

### Impact

- Testing context behavior requires mocking multiple interacting components simultaneously.
- A bug in `mission_tracker` can silently corrupt context injection (e.g., marking sub-tasks
  satisfied when they are not, causing the planner to skip steps).
- Adding new context-aware features requires understanding the implicit ordering in `planner_node.py`.

### Proposed fix (Phase 8.2)

Designate `ContextManager` as the **single owner** of all `mission_contexts` state transitions.
Add a new method `on_subtask_complete(mission_id, subtask_id, tool_results)` to `ContextManager`.
Refactor `mission_tracker.refresh_mission_status()` to call `context_manager.on_subtask_complete()`
rather than mutating `state["missions"]` directly. Document the ownership contract in a module
docstring at the top of `context_manager.py`.

## Proposed Architectural Changes (Phase 8.2 Scope)

Four concrete changes, each addressing one failure mode:

### Change 1: SubTaskAwareToolResult ŌĆö `on_tool_result()` signature extension

- **What:** Add `sub_task_id: str | None = None` parameter to `on_tool_result()`.
- **How:** `planner_node.py` resolves the active sub-task from `mission_tracker` before calling
  `on_tool_result()`. The sub-task ID is forwarded. Backward-compatible: existing callers omit
  the parameter and results land in the `"_unaffiliated"` bucket.
- **Scope:** `context_manager.py` (signature), `planner_node.py` (call site), `mission_tracker.py`
  (active sub-task accessor).

### Change 2: `sub_task_results` dict on `MissionContext`

- **What:** Replace `key_results: dict[str, str]` with `sub_task_results: dict[str, dict[str, str]]`.
  Outer key: sub-task ID (or `"_unaffiliated"`). Inner key: tool name. Inner value: result summary.
- **How:** Update `on_tool_result()`, `extract_artifacts()`, `build_summary()`, and any test
  fixtures that construct `MissionContext` directly.
- **Migration:** `key_results` removed. Existing persisted contexts in `mission_contexts` store
  will lack `sub_task_results` ŌĆö deserialize with empty dict default.

### Change 3: `get_injection_state()` observable API

- **What:** New read-only method on `ContextManager` returning `InjectionState` dataclass.
- **How:** Pure read ŌĆö no mutation. `InjectionState` fields derived from existing instance vars
  (`_injected_mission_ids`, `_last_injection_chars`, `_injected_cross_run_keys`).
- **Scope:** `context_manager.py` only. No callers change behavior ŌĆö tests gain assertion surface.

### Change 4: Single-owner mutation contract via `on_subtask_complete()`

- **What:** New `on_subtask_complete(mission_id, subtask_id, tool_results)` method on
  `ContextManager`. `mission_tracker.refresh_mission_status()` delegates sub-task completion
  recording to this method.
- **How:** Remove direct `state["missions"]` mutation from `mission_tracker` for sub-task
  completion events. `mission_tracker` retains responsibility for *detecting* satisfaction;
  `ContextManager` takes responsibility for *recording* it.
- **Scope:** `context_manager.py` (new method), `mission_tracker.py` (delegation refactor),
  `planner_node.py` (remove redundant mutation guards).

## Success Criteria for Phase 8.2

The redesign is complete when all of the following hold:

1. **Sub-task attribution preserved through eviction:** After `on_mission_complete()` evicts
   messages, `build_planner_context_injection()` emits lines of the form
   `"[sub-task 1.2] sort_array ŌåÆ [1, 2, 3]"` ŌĆö the planner can identify which sub-task produced
   each cached result without parsing raw result text.

2. **No last-write-wins for repeated tool calls:** When two different sub-tasks call `data_analysis`
   in the same mission, both results appear in `sub_task_results` under separate sub-task keys.
   The auditor's `chain_integrity` check passes for both.

3. **`get_injection_state()` returns accurate state:** A test can call `ctx_mgr.get_injection_state()`
   after injection and assert `"mission-1"` is in `injected_mission_ids` ŌĆö without parsing
   `state["messages"]` content.

4. **`mission_tracker` tests are import-isolated:** The test file `tests/unit/test_mission_tracker.py`
   can run without importing `context_manager` (no `from context_manager import ...` in the
   module or its test fixtures).

5. **All existing tests pass:** The 823+ tests currently passing continue to pass after the
   changes. No regressions introduced by the schema and API changes.

6. **Integration smoke test:** A scripted multi-mission run where two missions share a tool name
   shows distinct result entries in the final `audit_report.mission_reports` ŌĆö no result
   overwriting detected.
