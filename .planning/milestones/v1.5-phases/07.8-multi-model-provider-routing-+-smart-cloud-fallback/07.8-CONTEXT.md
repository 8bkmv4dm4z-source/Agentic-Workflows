# Phase 07.8: Multi-Model Provider Routing + Smart Cloud Fallback - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend LlamaCppChatProvider to support alias-based routing for multi-model llama-server instances. Upgrade ModelRouter to infer task complexity from runtime signals (token_budget_remaining, mission type, retry count) instead of caller-supplied labels. Add automatic cloud fallback: when the local llama.cpp model fails validation (schema mismatch, repeated parse failures, timeout), auto-escalate to Groq API with retry chain tracking in structural_health.

</domain>

<decisions>
## Implementation Decisions

### Alias Routing Model
- Single llama-server with multiple --model/--alias flags (not separate server instances)
- `with_alias(alias) -> LlamaCppChatProvider` returns a NEW instance sharing the same OpenAI client but with different model name — safe for concurrent use in ModelRouter
- Configuration via separate env vars: `LLAMA_CPP_STRONG_ALIAS` and `LLAMA_CPP_FAST_ALIAS`; if unset, both use the same auto-detected model (current behavior preserved)
- `build_provider()` unchanged — returns single LlamaCppChatProvider as today
- Caller (graph.py `__init__`) reads alias env vars and calls `with_alias()` to create strong/fast providers for ModelRouter

### Routing Signals Design
- `route_by_intent()` replaced by `route_by_signals(signals: RoutingSignals)` — single entry point for signal-based routing
- `RoutingSignals` TypedDict: `token_budget_remaining`, `mission_type`, `retry_count`, `step`, plus `intent_classification: dict | None`
- Thresholds for strong vs fast: multi_step / retry>=2 / budget<5000 -> strong, else -> fast
- Routing decisions tracked in structural_health as breakdown dict: `routing_decisions: {'strong': N, 'fast': M}`

### Cloud Fallback Trigger + Behavior
- Triggers on BOTH: `ProviderTimeoutError` OR 2 consecutive parse failures (after 7.7 escalation chain exhausts hint->retry->accept)
- Default fallback provider: Groq (fast, free tier, already integrated)
- `LangGraphOrchestrator.__init__` accepts `fallback_provider: ChatProvider | None` — callers (run.py, app.py) build from env and pass in. If None, no fallback (current behavior preserved)
- Per-step fallback: each step independently tries local first, cloud fallback is NOT sticky across steps — minimizes cloud usage
- Events counted in structural_health: `cloud_fallback_count`

### Structural Health Expansion
- `cloud_fallback_count: int` — total cloud fallback events
- `local_model_failures: {'timeout': N, 'parse': M}` — breakdown by failure type (timeout vs parse)
- `routing_decisions: {'strong': N, 'fast': M}` — routing distribution
- Dedicated "Routing & Fallback" section in audit_report — visible in both run.py AND user_run.py audit panels

### Claude's Discretion
- Threshold values for routing signals (module constants vs env-configurable)
- How to source mission_type for RoutingSignals (intent classifier primary vs step count vs both)
- Internal implementation of with_alias() client sharing

</decisions>

<specifics>
## Specific Ideas

- Single llama-server multi-model setup: `llama-server --model planner.gguf --alias planner --model executor.gguf --alias executor`
- Groq is the natural fallback — fast inference, free tier, already has GroqChatProvider integrated
- Routing distribution in audit helps validate whether the router is making sensible strong/fast splits
- Failure type breakdown (timeout vs parse) helps diagnose whether the problem is model speed or output quality
- User wants routing/fallback stats in user_run.py too (not just run.py)

<code_context>
## Existing Code Insights

### Reusable Assets
- `LlamaCppChatProvider` in provider.py: already has `model` field, `_detect_llama_cpp_model()`, GBNF grammar enforcement — with_alias() builds on top
- `ModelRouter` in model_router.py: `route()`, `route_by_intent()`, `has_dual_providers` — route_by_signals() replaces route_by_intent()
- `ProviderTimeoutError` in provider.py: existing exception class for timeout detection
- `_RetryingProviderBase`: shared retry logic all providers inherit
- `structural_health` dict in RunState: 4 existing counters (json_parse_fallback, schema_mismatch, format_correction_hints, format_retries)
- `GroqChatProvider`: already integrated, constructor takes optional model param

### Established Patterns
- `build_provider()` returns single ChatProvider from env; callers wire into ModelRouter
- `structural_health` counters incremented inline in graph.py with `.get(key, 0) + 1` pattern
- `ensure_state_defaults()` uses `.setdefault()` for new structural_health keys
- `TYPE_CHECKING` import guard for optional dependencies
- Constructor param with `| None` default for optional components (fallback_provider matches this)

### Integration Points
- `graph.py __init__`: reads alias env vars, creates ModelRouter with dual providers via with_alias()
- `graph.py __init__`: accepts fallback_provider param, stores for use in _plan_next_action()
- `graph.py _plan_next_action()`: cloud fallback logic after timeout or 2 consecutive parse failures
- `model_router.py`: route_by_signals() replaces route_by_intent(), new RoutingSignals TypedDict
- `state_schema.py`: new structural_health keys in new_run_state() and ensure_state_defaults()
- `run.py` + `user_run.py`: build fallback_provider from env, pass to orchestrator; display routing section in audit panel

</code_context>
