# Phase 07.8: Multi-Model Provider Routing + Smart Cloud Fallback - Research

**Researched:** 2026-03-10
**Domain:** Provider routing, model aliasing, cloud fallback orchestration
**Confidence:** HIGH

## Summary

Phase 07.8 extends three existing components: `LlamaCppChatProvider` (alias-based routing), `ModelRouter` (signal-based complexity inference), and `LangGraphOrchestrator` (cloud fallback on local model failure). All three are well-scoped modifications to existing code with clear integration points. The existing test infrastructure (ScriptedProvider, TrackedScriptedProvider, parametrized unit tests) provides strong patterns to follow.

The key technical insight is that llama-server's OpenAI-compatible API routes requests based on the `model` field in the request body, matching against `--alias` or `model_alias` config values. Since `LlamaCppChatProvider` already passes `self.model` to `self.client.chat.completions.create()`, the `with_alias()` factory only needs to create a new instance with a different `model` field sharing the same `OpenAI` client.

Cloud fallback is architecturally straightforward: `ProviderTimeoutError` is already caught in `_plan_next_action()` (graph.py:1285), and parse failure counting already exists via `structural_health["json_parse_fallback"]`. The new behavior inserts a cloud-provider retry between the existing failure detection and the existing deterministic fallback path.

**Primary recommendation:** Implement in 4 focused waves: (1) LlamaCpp alias support + unit tests, (2) RoutingSignals-based ModelRouter + parametrized tests, (3) cloud fallback wiring in graph.py + integration test, (4) structural_health expansion + audit panel display.

### Pattern 1: with_alias() Factory
**What:** Create a new LlamaCppChatProvider sharing the OpenAI client but with a different model name.
**When to use:** When graph.py `__init__` creates dual providers from env aliases.
**Example:**
```python
# In provider.py on LlamaCppChatProvider
def with_alias(self, alias: str) -> LlamaCppChatProvider:
    """Return a new provider instance using a different model alias.

    Shares the same OpenAI client and configuration but routes
    requests to a different model loaded in llama-server.
    """
    clone = LlamaCppChatProvider.__new__(LlamaCppChatProvider)
    clone.timeout_seconds = self.timeout_seconds
    clone.max_retries = self.max_retries
    clone.retry_backoff_seconds = self.retry_backoff_seconds
    clone.model = alias
    clone.client = self.client  # Share the OpenAI client
    clone._grammar_enabled = self._grammar_enabled
    return clone
```

### Pattern 2: RoutingSignals TypedDict
**What:** Replace intent_classification-only routing with multi-signal routing.
**When to use:** Every `_generate_with_hard_timeout()` call.
**Example:**
```python
# In model_router.py
class RoutingSignals(TypedDict):
    token_budget_remaining: int
    mission_type: str  # from intent_classification or "unknown"
    retry_count: int
    step: int
    intent_classification: dict[str, Any] | None

# Thresholds as module constants
_BUDGET_STRONG_THRESHOLD = 5000
_RETRY_STRONG_THRESHOLD = 2

def route_by_signals(self, signals: RoutingSignals) -> ChatProvider:
    """Route based on runtime signals -- replaces route_by_intent()."""
    # High-retry or low-budget -> strong (needs reliability)
    if signals["retry_count"] >= _RETRY_STRONG_THRESHOLD:
        return self._strong
    if signals["token_budget_remaining"] < _BUDGET_STRONG_THRESHOLD:
        return self._strong
    # Multi-step missions -> strong
    mission_type = signals.get("mission_type", "unknown")
    if mission_type == "multi_step":
        return self._strong
    # Intent classification as tiebreaker
    intent = signals.get("intent_classification")
    if intent is not None:
        if intent.get("complexity") == "complex":
            return self._strong
        if intent.get("complexity") == "simple":
            return self._fast
    return self._fast
```

### Pattern 3: Cloud Fallback in _plan_next_action
**What:** After local model fails (timeout or 2 consecutive parse failures), retry on cloud provider before deterministic fallback.
**When to use:** In the existing ProviderTimeoutError catch block and after parse failure counting.
**Example:**
```python
# In graph.py _plan_next_action(), after ProviderTimeoutError catch
except ProviderTimeoutError as exc:
    if self._fallback_provider is not None:
        try:
            model_output = self._fallback_provider.generate(
                state["messages"], response_schema=self._action_json_schema
            )
            state["structural_health"]["cloud_fallback_count"] = (
                state["structural_health"].get("cloud_fallback_count", 0) + 1
            )
            state["structural_health"].setdefault("local_model_failures", {})
            state["structural_health"]["local_model_failures"]["timeout"] = (
                state["structural_health"]["local_model_failures"].get("timeout", 0) + 1
            )
            # Continue with model_output from cloud provider...
        except Exception:
            pass  # Fall through to existing deterministic fallback
    # ... existing timeout handling continues
```

### Anti-Patterns to Avoid
- **Sticky cloud fallback:** Do NOT make fallback sticky across steps. Each step tries local first, cloud only on failure. This minimizes cloud usage.
- **Modifying build_provider():** The CONTEXT.md explicitly says `build_provider()` is unchanged. Alias wiring happens in graph.py `__init__`, not in the provider factory.
- **Breaking route_by_intent() callers:** `route_by_intent()` must be replaced by `route_by_signals()`, but `_generate_with_hard_timeout()` is the only caller (graph.py:1696). Update that single call site.
- **Blocking on fallback provider construction:** If `GROQ_API_KEY` is not set, `GroqChatProvider()` raises `ValueError`. Run.py must catch this and pass `fallback_provider=None`.

| Problem | Don't Build | Use Instead | Why |

## Common Pitfalls

### Pitfall 1: with_alias() Missing Attributes
**What goes wrong:** `LlamaCppChatProvider.__new__()` bypasses `__init__`, so `_RetryingProviderBase.__init__()` never runs. Missing `timeout_seconds`, `max_retries`, `retry_backoff_seconds`.
**Why it happens:** Using `__new__` to avoid side effects (like `_detect_llama_cpp_model` HTTP call).
**How to avoid:** Explicitly copy ALL required attributes from the source instance: `timeout_seconds`, `max_retries`, `retry_backoff_seconds`, `model`, `client`, `_grammar_enabled`. Also copy `use_native_chat_api=False`, `native_client=None`, `native_chat_url=None` (the alias path always uses the OpenAI-compatible client, not native chat).
**Warning signs:** `AttributeError` in `_request_with_retries` when using the aliased provider.

### Pitfall 2: Consecutive Parse Failure Counter Scope
**What goes wrong:** Using `structural_health["json_parse_fallback"]` as the consecutive counter. This is a cumulative counter, not a per-step consecutive counter.
**Why it happens:** Confusing cumulative metrics with per-step state.
**How to avoid:** Track consecutive parse failures as a local variable or separate state field (e.g., `retry_counts["consecutive_parse_failures"]`), reset to 0 on successful parse. Trigger cloud fallback when this reaches 2.
**Warning signs:** Cloud fallback triggers after 2 total parse failures across the entire run instead of 2 consecutive.

### Pitfall 3: Fallback Provider Construction Failure in run.py
**What goes wrong:** `GroqChatProvider()` raises `ValueError("GROQ_API_KEY not found")` when the env var is not set, crashing run.py startup.
**Why it happens:** Not wrapping fallback provider construction in try/except.
**How to avoid:** Wrap in try/except, pass `fallback_provider=None` on failure. Log a warning.
**Warning signs:** `ValueError` on startup when GROQ_API_KEY is not configured.

### Pitfall 4: route_by_signals() Breaking Existing Tests
**What goes wrong:** Removing `route_by_intent()` breaks existing unit tests in `test_model_router_wiring.py`.
**Why it happens:** Tests directly call `route_by_intent()`.
**How to avoid:** Keep `route_by_intent()` as a deprecated wrapper that builds RoutingSignals from intent_classification, OR update all tests to use `route_by_signals()`. The CONTEXT.md says "replaced by" which implies removal -- update tests.
**Warning signs:** Test failures in `test_model_router_wiring.py` after removing `route_by_intent()`.

### Pitfall 5: Cloud Fallback Doubling Timeout
**What goes wrong:** Local provider times out after 45s, then cloud fallback adds another 30s, making total step time 75s+.
**Why it happens:** Cloud fallback runs after local timeout is already exceeded.
**How to avoid:** This is acceptable per design (per-step fallback, not sticky). Document the latency cost. Consider whether the hard timeout in `_generate_with_hard_timeout()` should encompass both attempts or just the local one. Recommendation: let cloud fallback run outside the hard timeout wrapper since it's a recovery path.
**Warning signs:** Very long step times when fallback triggers.

### Pitfall 6: RoutingSignals mission_type Extraction
**What goes wrong:** `mission_type` is buried in `intent_classification` dict (from IntentClassification.to_dict()), not directly available in state.
**Why it happens:** IntentClassification is stored in `structured_plan.intent_classification` as a dict.
**How to avoid:** Extract from `(state.get("structured_plan") or {}).get("intent_classification", {}).get("mission_type", "unknown")`. This is the existing pattern at graph.py:1041.
**Warning signs:** `mission_type` is always "unknown" because extraction path is wrong.

## Open Questions

1. **Routing decision tracking needs state access**
   - What we know: `_generate_with_hard_timeout()` doesn't receive `state` -- it only gets `messages`, `complexity`, `intent_classification`.
   - What's unclear: How to increment `routing_decisions` in structural_health without state access.
   - Recommendation: Either (a) pass state to `_generate_with_hard_timeout()`, (b) track routing decisions on `self._router` and flush to state periodically, or (c) track at the call site in `_plan_next_action()` where state is available. Option (c) is simplest -- route in `_plan_next_action` before calling generate.

2. **Consecutive parse failure tracking for cloud fallback trigger**
   - What we know: The 7.7 format correction chain already counts drift: hint (1st) -> retry (2nd) -> accept (3rd+). The CONTEXT says "2 consecutive parse failures after escalation chain exhausts."
   - What's unclear: Does "2 consecutive" mean 2 steps in a row with parse failures, or 2 format_retries within the escalation chain?
   - Recommendation: Track as a separate counter `_consecutive_parse_failures` on the orchestrator instance, increment when escalation chain reaches "accept" stage (i.e., drift_count >= 3), reset to 0 on clean parse. Trigger cloud fallback when counter hits 2.

3. **with_alias() and native chat API path**
   - What we know: LlamaCppChatProvider has two code paths: OpenAI-compatible client (`self.client`) and native chat API (`self.native_client`).
   - What's unclear: Should aliased providers support the native chat API?
   - Recommendation: No. The native chat API is for `OLLAMA_NUM_CTX` pass-through. Aliased providers always use the OpenAI-compatible path where `model` field routing works. Set `use_native_chat_api=False` on clones.
