# Phase 07.6: LLM Output Structure Stabilization - Context

**Gathered:** 2026-03-09 (PRD Express Path), **Updated:** 2026-03-10 (full refresh)
**Status:** Complete (16/16 verified + quick-5 extension)

<domain>
## Phase Boundary

This phase mechanically enforces LLM output structure for weak local models running on LlamaCpp via SYCL backend. It fixed the phi4 context overflow blocker (8192-token context), added provider-aware compact prompts, re-enabled GBNF grammar for llama-cpp, instrumented the fallback parser, converted handoff TypedDicts to Pydantic, persisted chunked-read cursors to survive context eviction, and added structural health metrics to the audit report.

**Post-phase extension (quick-5):** Added tool arg signatures to compact prompt (`classify_intent(text)` instead of bare `classify_intent`) and JSON Schema response_format on `ChatProvider.generate()` — LlamaCpp uses it when grammar is disabled, OpenAI uses it natively, Groq/Ollama ignore it.

**NOT in scope:** PlanAction schema (deferred — needs SLM validation after 7.7/7.8 stabilize routing and fallback).

</domain>

<decisions>
## Implementation Decisions

### Primary Runtime (Updated 2026-03-10)

- **LlamaCpp via SYCL** is the primary local runtime (not Ollama)
- **phi4** is the current reference model (context_size=8192, best results to date)
- After phases 7.7-7.8 improve routing and fallback, model experimentation planned (newer GGUF models, llama.cpp releases, Ollama-popular model equivalents)
- **Groq** is the cloud fallback provider (Phase 7.8)
- **OpenAI** is optional/secondary
- Ollama code remains in codebase but is not the active development target

### Provider-Aware Prompt Sizing (Obstacle 0 Fix)

- `context_size()` on all providers: LlamaCpp=env/8192, Groq=32768, OpenAI=128000, Ollama=env/32768
- `_select_prompt_tier()`: <=10000 tokens = compact, >10000 = full
- Compact prompt: COMPACT directive from supervisor.md, tool names with arg signatures, env block
- Full prompt: detailed arg signatures, codebase context injection, comprehensive rules

### Tool Schema Enforcement (Quick-5 Extension)

- `Tool.required_args()` parses description strings via regex to extract arg names/types
- Compact prompt lists `tool_name(arg1, arg2)` instead of bare names
- `_build_action_json_schema()` generates anyOf JSON schema from live tool registry
- `ChatProvider.generate(response_schema=dict|None)` protocol extension
- LlamaCpp: uses response_schema when grammar disabled (mutually exclusive with GBNF)
- OpenAI: uses response_schema if provided, else falls back to built-in format
- Groq: intentionally ignores response_schema — json_object mode sufficient for now
- Schema cached as `self._action_json_schema` at init time

### GBNF Grammar

- Enabled by default on LlamaCpp (`_grammar_enabled=True` unless `LLAMA_CPP_GRAMMAR=false`)
- Mutually exclusive with response_format in llama-server
- `.env.example` clarifies: grammar is ENABLED by default, set to 0 to disable

### Fallback Parser Instrumentation

- `parse_action_json(model_output, step=0) -> tuple[dict, bool]` — returns (data, used_fallback)
- `parse_all_actions_json(model_output) -> tuple[list[dict], bool]` — multi-action extraction
- WARNING log: `PARSER FALLBACK used=extract_first_json step=N model_output=<200 chars>`
- `validate_action()` normalizes `{"tool": "X", flat_args}` → `{"action": "tool", "tool_name": "X", "args": {...}}`

### Handoff Pydantic Conversion

- `TaskHandoff` and `HandoffResult`: Pydantic BaseModel with `extra="forbid"`
- Factory functions `create_handoff()` / `create_handoff_result()` preserved
- `.model_dump()` at RunState append sites for checkpoint serialization safety

### Sub-Task Cursor Persistence

- `upsert_cursor()` / `get_cursor()` / `get_active_cursors()` on MissionContextStore
- SQLite: in-memory dict (pool=None); Postgres: `sub_task_cursors` table (migration 005)
- After context eviction, cursor hints re-injected as `[Orchestrator]` messages
- `__cursor_resume=True` bypasses `seen_tool_signatures` check (read_file_chunk only)

### Structural Health Metrics

- `structural_health` dict in RunState: `json_parse_fallback`, `schema_mismatch` counters
- Exposed in `audit_report["structural_health"]` after every run
- Counters increment in planner parse path (graph.py)

### What's Locked (don't change in 7.7-7.9)

- Compact/full prompt tier system (`_select_prompt_tier()` threshold, tier selection at init)
- GBNF grammar enforcement on LlamaCpp
- structural_health counter infrastructure (add new counters, don't change existing)
- Cursor persistence (upsert_cursor/get_cursor/get_active_cursors)
- Pydantic handoffs with `extra="forbid"`
- `validate_action()` normalization (accepts flat format, normalizes to canonical)

### What's Open for Evolution (7.7-7.9 can change)

- **ModelRouter input signals** — currently explicit TaskComplexity labels, 7.7 feeds intent, 7.8 adds runtime signals
- **Tool arg declaration method** — regex-parsed `Tool.required_args()` is fragile and causes real failures; 7.7 should replace with typed `Tool.args_schema`
- **response_schema API** — `generate(response_schema=dict)` interface may need per-step or provider-specific schemas
- **Directive COMPACT/FEW_SHOT** — only supervisor.md has COMPACT; executor.md and evaluator.md need both COMPACT and FEW_SHOT (land in 7.7)
- **PLAN INVALID retry hint** — exists but effectiveness with SLMs unvalidated; 7.7 should test and refine or replace

### Foundation Decisions for Downstream Phases

#### Format Correction Escalation Chain (implement in 7.7)
Models (especially phi4) consistently produce `{"tool": "X", flat_args}` instead of canonical format. validate_action normalizes it, but the model doesn't learn. Implement progressive escalation:
1. **First drift**: Inject `[Orchestrator]` user-message hint with correct format example (free)
2. **Second consecutive drift**: Retry with explicit system-level correction (costs 1 LLM call)
3. **Third**: Accept normalized output and move on
4. **Track**: `structural_health["format_correction_hints"]` and `structural_health["format_retries"]` counters

#### Store-Then-Query Pattern for Complex Tasks (teach in 7.7, implement in 7.9)
For complex tasks like analyzing a 2000-line file:
- Planner should plan: outline_code first, then read_file_chunk in batches
- Each batch result gets **stored to Postgres** (not just in context window)
- When context window evicts earlier batches, planner can **query the DB** to recall them
- Context window = working set (cache), Postgres = full history (disk)
- 7.7: Few-shot examples teach the planner the store-then-query pattern
- 7.9: query_context tool + step-result storage infrastructure
- Storage backend: Claude's discretion (ArtifactStore extension or new table)

### Claude's Discretion

- SQLite cursor storage: in-memory dict on MissionContextStore instance (no disk table)
- Cursor key format: composite key for dedup
- `parse_action_json()` step parameter: threaded through callers as optional kwarg
- Backward compat: all changes additive — existing behavior unchanged when tier=full

### Compact Supervisor COMPACT Section (as-built)
```
## COMPACT
You emit exactly one JSON action per response. No markdown, no prose, no XML. Pure JSON only.
Schema: {"action":"tool","tool_name":"X","args":{...}} | {"action":"finish","answer":"X"} | {"action":"clarify","question":"X"}
Rules: Use python3 not python. Use run_bash for shell commands. One tool per response.
Example: {"action":"tool","tool_name":"run_bash","args":{"command":"python3 -c 'print(1)'","timeout":10}}
Example: {"action":"finish","answer":"Task complete: wrote fib50.txt with 50 numbers"}
```
