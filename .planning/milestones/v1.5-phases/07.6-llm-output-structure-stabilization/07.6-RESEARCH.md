## Summary

Phase 07.6 is a mechanical hardening pass with six loosely coupled sub-problems. The primary driver is the phi4 context overflow blocker: phi4 (and any small GGUF) runs llama-cpp with an 8192-token context window, but the current system prompt is roughly 2000–2500 chars of instruction text plus a full codebase context block — leaving only ~5000 tokens for conversation history, which overflows quickly on multi-mission runs. The fix is a two-tier prompt system gated on `provider.context_size()`.

The remaining work (cursor persistence, Pydantic handoffs, parser instrumentation, structural health metrics) is independently executable and cannot regress existing behavior because all changes are additive or replace internal-only contracts.

The codebase is in very good shape for this phase: every affected module is clean, well-typed, and already has a test file. The main risk is the Pydantic handoff migration breaking test_handoff.py which uses TypedDict-style dict key access — those tests need updating alongside the production change.

**Primary recommendation:** Implement in the locked order (C0/C1/C2/C3 first) so phi4 runs are unblocked before any other work is committed.

## Architecture Patterns

### Pattern 1: Protocol Extension — Adding `context_size()` to ChatProvider

**What:** Add `context_size() -> int` as a new method in the `ChatProvider` Protocol in `provider.py`. Each of the four concrete classes gets an implementation.

**Current Protocol (provider.py:66-70):**
```python
class ChatProvider(Protocol):
    """Provider contract used by the LangGraph planner node."""
    def generate(self, messages: Sequence[AgentMessage]) -> str: ...
```

**After change:**
```python
class ChatProvider(Protocol):
    def generate(self, messages: Sequence[AgentMessage]) -> str: ...
    def context_size(self) -> int: ...
```

**Implementations:**
- `LlamaCppChatProvider.context_size()`: `return int(os.getenv("LLAMA_CPP_N_CTX", "8192"))` — the env var already exists in the project (referenced in comments)
- `GroqChatProvider.context_size()`: `return 32768`
- `OpenAIChatProvider.context_size()`: `return 128000`
- `OllamaChatProvider.context_size()`: `return self.num_ctx if self.num_ctx > 0 else int(os.getenv("OLLAMA_NUM_CTX", "32768"))` — `self.num_ctx` is already populated in `__init__` from `_env_int("OLLAMA_NUM_CTX", 0)`

**Impact on callers:** `build_provider()` returns a `ChatProvider` — since Protocol uses structural subtyping, any caller holding a `ChatProvider` can call `.context_size()` without import changes. The `ScriptedProvider` in tests also needs `context_size() -> int` added (return a large value like 32768 to stay in `full` tier and avoid breaking test expectations).

### Pattern 3: Handoff Pydantic Migration

**Current handoff.py structure:**
- `TaskHandoff(TypedDict)` — 6 fields
- `HandoffResult(TypedDict)` — 5 fields
- `create_handoff()` — returns `TaskHandoff(...)` (TypedDict constructor call)
- `create_handoff_result()` — returns `HandoffResult(...)` (TypedDict constructor call)

**Current test_handoff.py access pattern (lines 18–60):** Uses `h["task_id"]`, `h["specialist"]`, `h["tool_scope"]` — TypedDict dict-key access. Pydantic models support `model["key"]` via `__getitem__` ONLY if you do NOT set `model_config`. By default, Pydantic BaseModel does NOT support subscript access. Two options: (a) keep dict access by adding `__getitem__` to the model, or (b) update tests to use `h.task_id`. The CONTEXT.md decision says to keep factory functions and use `model_dump()` for RunState — the cleanest approach is update tests to attribute access.

**After migration:**

**RunState storage:** Wherever `create_handoff()` / `create_handoff_result()` results are appended to `handoff_queue` / `handoff_results` (which are `list[dict[str, Any]]`), the Pydantic objects must be serialized: `.model_dump()`. **All callers of `create_handoff` / `create_handoff_result` in production code:** `graph.py` (1 file). Test files: `test_handoff.py`. Both need updating.

## Common Pitfalls

### Pitfall 1: Protocol Structural Compatibility — ScriptedProvider Missing `context_size()`

**What goes wrong:** `ScriptedProvider` in `tests/conftest.py` implements the `ChatProvider` Protocol by only having `generate()`. After `context_size()` is added to the Protocol, `ScriptedProvider` no longer satisfies it — mypy raises an error, and `LangGraphOrchestrator.__init__` calling `provider.context_size()` at runtime raises `AttributeError` for tests that inject `ScriptedProvider`.

**Why it happens:** Python Protocol structural subtyping is checked at runtime (via mypy/pyright) but AttributeError only surfaces at call time.

**How to avoid:** Add `def context_size(self) -> int: return 32768` to `ScriptedProvider` before any other change. Check `tests/conftest.py` for all mock providers.

**Warning signs:** `AttributeError: 'ScriptedProvider' object has no attribute 'context_size'` in test runs.

### Pitfall 2: Pydantic Model Subscript Access in Tests

**What goes wrong:** `test_handoff.py` uses `h["task_id"]` syntax (TypedDict-style dict subscript). Pydantic BaseModel does not support `model["key"]` by default — it raises `TypeError: 'TaskHandoff' object is not subscriptable`.

**Why it happens:** TypedDict instances ARE dicts; Pydantic BaseModel instances are NOT dicts unless you explicitly implement `__getitem__`.

**How to avoid:** Update all `test_handoff.py` accesses to attribute access (`h.task_id`, `h.specialist`, etc.) in the same PR as the production change. Also verify graph.py call sites that append handoffs to state — they must call `.model_dump()` before appending.

**Warning signs:** `TypeError` in test_handoff.py.

### Pitfall 3: Cursor Bypass Logic Must Not Bypass All Duplicate Detection

**What goes wrong:** The `__cursor_resume: True` flag in the action dict bypasses `seen_tool_signatures`. If this bypass is not narrowly scoped to `read_file_chunk` calls with an active cursor, it could allow any arbitrary `read_file_chunk` call to bypass deduplication — opening a loop hole.

**Why it happens:** The bypass flag is trusted if set on the action dict. Any code path that sets `__cursor_resume` on a non-cursor action would bypass deduplication.

**How to avoid:** Only set `__cursor_resume` when: (a) tool_name == "read_file_chunk" AND (b) an active cursor exists for the given path in `get_active_cursors()`. The bypass check in graph.py should verify both conditions.

**Warning signs:** Infinite `read_file_chunk` loops not caught by deduplication.

### Pitfall 4: `structural_health` Counter Increments From action_parser Without State Access

**What goes wrong:** `parse_action_json()` and `validate_action()` in `action_parser.py` have no access to `state`. Trying to increment `state["structural_health"]["json_parse_fallback"]` from inside action_parser creates a circular dependency or forces state into a module-level global.

**Why it happens:** action_parser.py is a pure parsing module with no state dependency — this is by design.

**How to avoid:** Track counter increments at the graph.py call sites: the planner node catches the fallback path (via return flag or exception subtype) and increments the state counter. Never import state into action_parser.py.

### Pitfall 5: Compact Prompt Breaks Existing Tool-Calling Tests

**What goes wrong:** Tests that assert specific system prompt content (e.g., checking for tool arg signatures in the prompt) will fail when running with a mock provider that returns `context_size() = 8192` (compact tier).

**Why it happens:** If `ScriptedProvider.context_size()` returns 8192 (phi4-sized), all test runs will get compact tier, breaking prompt-content tests.

**How to avoid:** Set `ScriptedProvider.context_size()` to return 32768 (full tier). Only llama-cpp environments get compact tier in practice.

### MissionContextStore cursor methods (in-memory fallback)

def __init__(self, pool=None, embedding_provider=None):
    self._pool = pool
    self._embedding_provider = embedding_provider
    self._cursors: dict[str, dict] = {}  # in-memory cursor store for SQLite/test path

def upsert_cursor(self, run_id, plan_step_id, mission_id, tool_name, key, cursor, total):
    if self._pool is None:
        cursor_key = f"{run_id}:{mission_id}:{tool_name}:{key}"
        self._cursors[cursor_key] = {"next_offset": cursor, "total": total,
                                      "tool_name": tool_name, "key": key,
                                      "run_id": run_id, "mission_id": mission_id}
        return

def get_cursor(self, run_id, plan_step_id, mission_id, tool_name, key):
    if self._pool is None:

        entry = self._cursors.get(cursor_key)
        return entry["next_offset"] if entry else None

def get_active_cursors(self, run_id):
    if self._pool is None:
        return [v for v in self._cursors.values() if v.get("run_id") == run_id]

def parse_action_json(model_output: str, step: int = 0) -> dict[str, Any]:
    cleaned = _strip_thinking(model_output)
    try:
        data = json.loads(cleaned)
        if not isinstance(data, dict):
            raise ValueError("action payload must be a JSON object")
        return data
    except json.JSONDecodeError as exc:
        candidate = extract_first_json_object(cleaned)
        if not candidate:
            raise ValueError(f"invalid json: {str(exc)}") from exc
        # Fallback path — emit structured warning
        prefix = cleaned[:cleaned.find("{")] if "{" in cleaned else ""
        _LOG.warning(
            "PARSER FALLBACK used=extract_first_json step=%d model_output=%.200s",
            step, model_output,
        )
        if prefix.strip():
            _LOG.warning("PARSER FALLBACK prose_prefix=%.200s", prefix.strip())
        try:
            recovered = json.loads(candidate)
        except json.JSONDecodeError as recover_exc:
            raise ValueError(f"invalid json: {str(recover_exc)}") from recover_exc
        if not isinstance(recovered, dict):
            raise ValueError("action payload must be a JSON object") from None
        return recovered
