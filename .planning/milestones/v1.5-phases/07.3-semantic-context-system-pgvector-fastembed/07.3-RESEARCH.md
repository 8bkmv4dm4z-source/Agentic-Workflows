## Summary

Phase 7.3 adds a 5-layer hybrid retrieval system that persists completed mission contexts across process restarts and horizontal replicas. The architecture is straightforward to integrate because the existing codebase already uses psycopg + ConnectionPool (sync, `%s` placeholders, `with pool.connection() as conn:` pattern), pgvector extension is already enabled in `002_foundation.sql`, and `ContextManager` is a plain class with no metaclass complexity that accepts new optional parameters cleanly.

The critical findings are: (1) `vector(1536)` appears in `file_chunks` and `solved_tasks` tables in `002_foundation.sql` — these must be migrated to `vector(384)` or the HNSW index creation at that dimension will fail; (2) `ContextManager.__init__` currently has exactly 3 params (`large_result_threshold`, `sliding_window_cap`, `step_threshold`) with no `*` forcing keyword-only — adding new optional params is straightforward; (3) `LangGraphOrchestrator.__init__` already uses keyword-only args (has `*` at position 1) and constructs `ContextManager()` at line 187 with zero arguments, making the wiring change minimal; (4) the tool registry has exactly 36 registered tool names, giving a clean TOOL_BITS mapping that fits in a 64-bit int; (5) the existing Postgres test pattern (`pytest.importorskip("psycopg_pool")` + `@requires_postgres` + `pg_pool`/`clean_pg` fixtures) is the mandatory pattern for new storage tests.

**Primary recommendation:** Follow the codebase patterns exactly — psycopg sync `%s`-style SQL, `ConnectionPool` passed via constructor, `pytest.importorskip` guard, TypedDict for result types (matches `RunResult` style). The context package and storage modules are independent and can be developed in parallel with the ContextManager integration.

## Common Pitfalls

### Pitfall 1: vector(1536) Migration Scope
**What goes wrong:** `002_foundation.sql` has `vector(1536)` in two tables (`file_chunks.embedding` and `solved_tasks.embedding`). If these tables already have data in Postgres and you try to `ALTER COLUMN` to `vector(384)`, it fails without `USING` cast. If the migration runs before pgvector extension, `vector` type is unknown.
**Why it happens:** 002_foundation.sql creates tables that were placeholders — no code writes to them yet. But if an existing Postgres instance already ran 002, the schema is set.
**How to avoid:** Migration 002 update should use `ALTER TABLE ... ALTER COLUMN embedding TYPE vector(384) USING embedding::text::vector(384)` or `DROP TABLE IF EXISTS + CREATE TABLE`. Since no code writes these tables yet, DROP+CREATE is safe. Document this in the migration comment.
**Warning signs:** `ERROR: column "embedding" is of type vector(1536) but expression is of type vector(384)`

### Pitfall 2: fastembed Optional Import
**What goes wrong:** Importing `fastembed` at module level in `embedding_provider.py` causes `ImportError` in CI (and for users without `[context]` extra).
**Why it happens:** fastembed is an optional dependency in `[context]` group.
**How to avoid:** In `FastEmbedProvider`:
```python
class FastEmbedProvider:
    def __init__(self, model_name: str = "BAAI/bge-small-en-v1.5") -> None:
        try:
            from fastembed import TextEmbedding
        except ImportError as e:
            raise ImportError("Install fastembed: pip install 'agentic-workflows[context]'") from e
        self._model = TextEmbedding(model_name)
```
**Warning signs:** `ImportError: No module named 'fastembed'` at collection time in CI.

### Pitfall 3: MockEmbeddingProvider Dimension Consistency
**What goes wrong:** Mock returns wrong-dimension vectors. Postgres will reject INSERT with `ERROR: expected 384 dimensions, not N`.
**Why it happens:** Mock uses SHA-256 (32 bytes = 256 bits). Must upsample to 384 floats deterministically.
**How to avoid:** Use `hashlib.sha256(text.encode()).digest()` (32 bytes), then expand to 384 floats via tiling or deterministic seeding with `random.seed(hash_bytes)` + `random.gauss()` x384, normalize to unit sphere.
**Warning signs:** pgvector dimension mismatch error on first INSERT attempt.

### Pitfall 4: build_planner_context_injection Cap Change Side Effect
**What goes wrong:** Bumping the 800-char cap to 1500 in `build_planner_context_injection` may affect existing tests that assert on the injected string length.
**Why it happens:** Existing tests for `build_planner_context_injection` may not mock the cap.
**How to avoid:** Search existing tests for `build_planner_context_injection` and `800` before changing. The current implementation doesn't enforce a char cap in code (it just logs "CONTEXT INJECT ... chars=...") — the 800 is a log/comment reference from MEMORY.md, not a hard code limit. Verify in context_manager.py before assuming a cap exists.
**Warning signs:** Test assertion failures on context injection string content.

### Pitfall 5: clean_pg Fixture Missing New Tables
**What goes wrong:** The existing `clean_pg` fixture truncates `graph_checkpoints, runs, memo_entries` only. Integration tests for new tables will fail with leftover rows between tests.
**Why it happens:** `clean_pg` in conftest.py has a hardcoded table list (line 96).
**How to avoid:** Either extend `clean_pg` to add `mission_contexts, mission_artifacts` OR create a new `clean_pg_full` fixture for Phase 7.3 tests. Preferred: extend the existing fixture since it's session-scoped with per-test truncation.
**Warning signs:** Integration tests polluting each other with stale rows.

### Pitfall 6: BIT(384) String Encoding
**What goes wrong:** pgvector BIT type requires a binary string `'010110...'` (384 chars of '0'/'1'), not a Python bytes object or integer.
**Why it happens:** Python `int` bitmask encoding doesn't map to Postgres BIT literal format.
**How to avoid:** Convert float vector to BIT string as:
```python
def _float_vec_to_bit_string(vec: list[float]) -> str:
    return "".join("1" if v >= 0.0 else "0" for v in vec)
```
Pass as plain string to psycopg. Verify with `SELECT B'010...'::bit(384)` in Postgres.
**Warning signs:** `psycopg.errors.InvalidTextRepresentation` on INSERT.

### Pitfall 7: HNSW Index Creation on Empty vs Populated Table
**What goes wrong:** HNSW index creation on a table with millions of rows is slow (minutes). On an empty table it's instant.
**Why it happens:** Index build scans all rows.
**How to avoid:** Create HNSW index in migration while table is empty. For production with existing data, create `CONCURRENTLY`.
**Warning signs:** Migration timeout in CI if table has data.

### Pitfall 8: on_mission_complete Async Context
**What goes wrong:** `on_mission_complete` is called from sync graph nodes (the entire graph runs synchronously). If `MissionContextStore._persist` uses `async`, it can't be called directly.
**Why it happens:** Graph nodes are sync; the existing `on_mission_complete` in context_manager.py is sync.
**How to avoid:** `MissionContextStore` persistence method MUST be synchronous — use `with self._pool.connection() as conn: conn.execute(...)` directly (same pattern as `PostgresCheckpointStore.save()`). Do NOT use `anyio.to_thread.run_sync` inside the store — that's only for async callers in the FastAPI layer.
**Warning signs:** `RuntimeError: no running event loop` when calling async from sync.

## Code Examples

### Cascade Query Structure (SQL)
```sql
-- L0: exact SHA-256 hit (short-circuit)
SELECT * FROM mission_contexts WHERE goal_hash = %s AND status = 'completed' LIMIT 1;

-- L1: tool bitmask structural match (short-circuit)
SELECT * FROM mission_contexts
WHERE (tool_pattern & %s) = %s AND status = 'completed'
ORDER BY created_at DESC LIMIT 10;

-- L2: BM25 full-text (tsvector ts_rank)
SELECT id, ts_rank(goal_tsvector, query) AS rank
FROM mission_contexts, plainto_tsquery('english', %s) AS query
WHERE goal_tsvector @@ query AND status = 'completed'
ORDER BY rank DESC LIMIT 20;

-- L4: cosine similarity HNSW
SELECT id, 1 - (embedding <=> %s::vector) AS score
FROM mission_contexts
WHERE status = 'completed'
ORDER BY embedding <=> %s::vector
LIMIT 20;
```

### RRF Fusion
```python
# Source: CONTEXT.md specification
def reciprocal_rank_fusion(*ranked_lists: list[str], k: int = 60) -> dict[str, float]:
    scores: dict[str, float] = {}
    for ranked in ranked_lists:
        for rank, doc_id in enumerate(ranked):
            scores[doc_id] = scores.get(doc_id, 0.0) + 1.0 / (k + rank + 1)
    return dict(sorted(scores.items(), key=lambda x: x[1], reverse=True))
```

### MockEmbeddingProvider (deterministic 384-dim)
```python
import hashlib
import random

class MockEmbeddingProvider:
    dimensions: int = 384

    def embed_sync(self, text: str) -> list[float]:
        seed = int.from_bytes(hashlib.sha256(text.encode()).digest(), "big")
        rng = random.Random(seed)
        vec = [rng.gauss(0, 1) for _ in range(self.dimensions)]
        norm = sum(v * v for v in vec) ** 0.5
        return [v / norm for v in vec] if norm > 0 else vec

    def embed(self, texts: list[str]) -> list[list[float]]:
        return [self.embed_sync(t) for t in texts]
```
