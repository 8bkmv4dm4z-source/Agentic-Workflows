# Phase 7.1: Context Manipulation for Better Sub-Agent Multi-Task Handling - Context

**Gathered:** 2026-03-07
**Status:** Ready for planning

<domain>
## Phase Boundary

Redesign how context (message history, tool results, mission state) is managed during multi-mission orchestration runs. Introduce a typed MissionContext model as the durable queryable store, implement event-driven context eviction, enrich specialist subgraph context, and scope planner messages per-mission. Actual parallel execution via Send() is out of scope — the context model should support it but wiring is deferred.

</domain>

<decisions>
## Implementation Decisions

### Mission Context Scoping
- On mission completion, summarize all of Mission N's tool result messages into a compact summary and replace them in the planner's message history
- Summary includes: mission goal, tools used, key results, and **cross-mission artifacts** (file paths, computed values, URLs discovered during execution that later missions may need)
- Artifact extraction is deterministic (parsed from tool result JSON, no LLM calls)
- Summaries injected as `role="user"` messages with `[Orchestrator]` prefix — avoids Ollama consecutive-system-message breakage (Phase 5 decision)
- Eviction trigger timing: Claude's discretion (event-driven at mission boundaries vs also mid-mission threshold)

### Specialist Context Enrichment
- Specialist subgraphs receive **mission goal + prior results summary** when invoked (not just tool_name/args/step)
- New **top-level fields** on ExecutorState and EvaluatorState TypedDicts (not crammed into input_context dict)
- Prior results summary uses the same format as mission boundary summaries — one abstraction, tested once
- Whether enriched context affects specialist behavior or is purely informational: Claude's discretion (likely informational given tools are deterministic)

### Context Eviction Strategy
- **Replace** the existing Ollama-specific `_evict_tool_result_messages()` entirely with a provider-agnostic eviction system
- **Event-driven eviction** with sliding window safety net:
  - Mission completion: summarize and prune completed mission messages
  - Large tool result received (>threshold): store full result in MissionContext, replace message with compact placeholder
  - Step threshold within a mission: compress intermediate planning messages
  - Sliding window hard cap (e.g., 30 messages): oldest non-system messages drop regardless of events
- Summaries are **deterministic** — extract key fields from tool result JSON programmatically, no LLM summarization
- **Full observability**: every eviction event emits both a log entry and a Langfuse trace event

### MissionContext Model
- **Typed Pydantic model** (`MissionContext`) stored as `dict[int, MissionContext]` on RunState
- Queryable context store pattern: planner queries MissionContext for what it needs, executes, updates context, messages get pruned
- **Hierarchical sub-mission support**: MissionContext supports nested sub-mission contexts (sub_mission_id level), matching structured plan architecture
- Context is checkpointed automatically via RunState (no new external store)
- **Isolated + explicit artifacts** for cross-mission data flow: Claude's discretion on exact isolation model

### Claude's Discretion
- Exact MissionContext Pydantic field names and structure (within the hierarchical sub-mission constraint)
- Whether eviction activates mid-mission or only at mission boundaries
- Whether specialist enriched context is informational-only or behavior-affecting
- Cross-mission isolation model (fully isolated + artifacts vs read-only cross-mission)
- MissionContext persistence approach (RunState field recommended, Claude confirms)
- Sliding window threshold values and large-tool-result size threshold

</decisions>

<specifics>
## Specific Ideas

- "Context as database, not as chat history" — the planner queries MissionContext for what it needs rather than accumulating everything in messages. This prevents context window explosion at high iteration counts.
- Cross-mission artifacts example: a file path discovered by write_file in Mission 1 that Mission 2's read_file needs — extracted deterministically from tool results and carried in MissionContext.
- The MissionContext schema should support a nested structure like: `{mission_id: {goal: str, artifacts: dict, sub_missions: {sub_id: {goal: str, result: ...}}}}` (exact Pydantic modeling at Claude's discretion).

</specifics>

<deferred>
## Deferred Ideas

- **Vector similarity for mission/result approximation** — using a vector library to find similar past missions or results to inform current mission execution. New retrieval capability, not context manipulation. Future phase.
- **Parallel mission execution via Send()** — actual fan-out wiring using LangGraph Send() API. Context model designed to support it but wiring is deferred (v2 requirements PRLL-01/PRLL-02).
- **LLM-generated summaries** — using a fast model for richer context summarization instead of deterministic extraction. Adds latency/cost; evaluate after deterministic approach is proven.

</deferred>
