# Phase 07.7: Hybrid Intent Classifier + Few-Shot Prompt Refinement - Research

**Researched:** 2026-03-10
**Domain:** LLM intent classification, few-shot prompt engineering, token budget management, format correction
**Confidence:** HIGH

## Summary

Phase 07.7 adds four interconnected capabilities to the orchestration engine: (1) an LLM-based intent classifier that runs inside `parse_missions()` to classify mission complexity before planning, (2) few-shot JSON examples in all three directive files with COMPACT sections for executor.md and evaluator.md, (3) per-role token budgets with a `_estimate_prompt_tokens()` helper and tool description truncation, (4) a format correction escalation chain (hint -> retry -> accept) that replaces the current all-or-nothing retry pattern, plus (5) typed `Tool.args_schema` replacing the fragile regex-based `required_args()`.

The codebase is well-prepared for these changes. `parse_missions()` already has a timeout pattern (threading + queue). `_build_system_prompt()` already reads `## COMPACT` sections from directives. `structural_health` dict already tracks `json_parse_fallback` and `schema_mismatch` counters. `validate_action()` already normalizes non-canonical formats. The `ModelRouter` currently only routes based on explicit `TaskComplexity` literals -- intent classification will replace this with data-driven routing.

**Primary recommendation:** Implement as 4 plans: (1) intent classifier + IntentClassification dataclass in mission_parser, (2) few-shot sections in directives + COMPACT sections + injection logic, (3) per-role token budgets + Tool.args_schema + _estimate_prompt_tokens, (4) format correction escalation chain + intent-driven ModelRouter wiring.

### Pattern 1: IntentClassification Dataclass

**What:** A simple dataclass stored in `StructuredPlan.intent_classification`
**When to use:** After deterministic parsing, before planning begins

```python
@dataclass
class IntentClassification:
    complexity: Literal["simple", "complex"]
    mission_type: str  # e.g., "file_io", "computation", "analysis", "multi_step"
    confidence: float  # 0.0-1.0 from LLM or heuristic
    source: str  # "llm" or "deterministic_fallback"

    def to_dict(self) -> dict[str, Any]:
        return {
            "complexity": self.complexity,
            "mission_type": self.mission_type,
            "confidence": self.confidence,
            "source": self.source,
        }
```

**Integration:** `StructuredPlan` gets a new field `intent_classification: IntentClassification | None = None`. The `to_dict()` and `from_dict()` methods must be updated accordingly.

### Pattern 2: Classifier Call Inside parse_missions()

**What:** Optional ChatProvider param; after deterministic parse completes, one quick LLM call classifies intent
**Timeout:** 500ms using same threading+queue pattern already in parse_missions()

```python
def parse_missions(
    user_input: str,
    timeout_seconds: float = 5.0,
    max_plan_steps: int = 7,
    classifier_provider: ChatProvider | None = None,  # NEW
    classifier_timeout: float = 0.5,  # NEW
) -> StructuredPlan:
    # ... existing parsing ...
    plan = _enforce_step_limit(plan, max_plan_steps)
    if classifier_provider is not None:
        plan.intent_classification = _classify_intent(
            user_input, plan, classifier_provider, classifier_timeout
        )
    else:
        plan.intent_classification = _deterministic_classify(plan)
    return plan
```

**Key detail:** The deterministic fallback should use step count + keyword complexity heuristics:
- 1-2 steps with simple tool keywords -> "simple"
- 3+ steps or complex keywords (analyze, compare, generate) -> "complex"
- Any regex_fallback parsing method -> "complex" (parser couldn't understand structure)

### Pattern 4: Format Correction Escalation Chain

**What:** Three-step progressive correction in `_plan_next_action()` when model output fails validation
**Where:** In the `except Exception` block at graph.py:1250+ where `invalid_json` retries are tracked

Current flow: increment invalid_count, inject retry hint, fail after max retries.

New flow:
1. **Step 1 (first drift):** Inject `[Orchestrator]` user-message hint with correct format example (free, no LLM call). Increment `format_correction_hints` counter.
2. **Step 2 (second consecutive drift):** Retry with explicit system-level correction (costs 1 LLM call). Increment `format_retries` counter.
3. **Step 3 (third consecutive drift):** Accept normalized output from `validate_action()` + log WARNING. No more retries.

**Key:** Track `consecutive_format_drift` in `retry_counts`. Reset to 0 when a correctly-formatted response arrives (at graph.py:1052 where `consecutive_empty` is reset).

### Pattern 5: Tool.args_schema Property

**What:** Replace regex-based `required_args()` with a typed `args_schema` dict
**Current fragile pattern (base.py:12-27):**
```python
def required_args(self) -> list[str]:
    # Parses "Required args: path (str), content (str)" from description
    m = re.search(r"Required args?:\s*(.+?)(?:\.|Optional|$)", desc, re.IGNORECASE)
```

**Problem:** Only ~5 of 36 tools include "Required args:" in their description. The regex returns `[]` for most tools, making `_build_action_json_schema()` produce empty args schemas and compact prompt signatures show just tool names without args.

**Solution:**
```python
class Tool:
    name: str
    description: str
    _args_schema: dict[str, dict[str, str]] | None = None  # Override in subclass

    @property
    def args_schema(self) -> dict[str, dict[str, str]]:
        """Return typed arg schema. Override _args_schema in subclass."""
        if self._args_schema is not None:
            return self._args_schema
        # Fall back to legacy regex parsing for backward compat
        return {arg: {"type": "string"} for arg in self.required_args()}
```

Each tool subclass declares `_args_schema` as a class variable:
```python
class WriteFileTool(Tool):
    name = "write_file"
    description = "Writes content to a file. Overwrites if exists."
    _args_schema = {
        "path": {"type": "string", "required": True},
        "content": {"type": "string", "required": True},
    }
```

**Consumers updated:**
- `_build_action_json_schema()` at graph.py:279: `tool.args_schema` instead of `tool.required_args()`
- `_tool_sig()` at graph.py:363-365: extract required arg names from `args_schema`

### Anti-Patterns to Avoid
- **Mixing classifier timeout with parse_missions timeout:** Keep them separate. The 5s parse timeout wraps deterministic parsing. The 0.5s classifier timeout wraps the LLM call independently.
- **Injecting few-shot on compact tier by default:** The compact tier exists because context is limited. Be conservative -- recommend full-only for few-shot, with compact getting at most 1 minimal example per role.
- **Using tiktoken for token estimation:** tiktoken adds a dependency and is model-specific. The project already uses `len//4` estimation (graph.py:1058). Stay consistent.

## Common Pitfalls

### Pitfall 1: StructuredPlan Serialization
**What goes wrong:** Adding `intent_classification` to StructuredPlan breaks `to_dict()`/`from_dict()` and checkpoint serialization
**Why it happens:** StructuredPlan is stored in `state["structured_plan"]` as a dict via `to_dict()`. If `from_dict()` doesn't handle the new field, deserialization fails.
**How to avoid:** Update both `to_dict()` and `from_dict()` to handle `intent_classification`. Use `None` as default for backward compat with existing checkpoint data.
**Warning signs:** Existing test_mission_parser tests fail

### Pitfall 2: Classifier Provider is the SAME Provider
**What goes wrong:** The classifier call uses the same provider as planning, but at init time (before `_generate_with_hard_timeout` is set up)
**Why it happens:** `parse_missions()` is called at graph.py:614 inside `prepare_state()`. The provider is available as `self.provider`.
**How to avoid:** Pass `self.provider` as `classifier_provider` at graph.py:614. The 500ms timeout is handled inside `_classify_intent()` itself, not via `_generate_with_hard_timeout`.

### Pitfall 3: ModelRouter Backward Compatibility
**What goes wrong:** Changing `ModelRouter.route()` signature breaks existing callers
**Why it happens:** `route()` is called at graph.py:1579/1585 with `TaskComplexity` string literal
**How to avoid:** Add optional `intent_classification` parameter to `route()`. When provided AND complexity is not in `_STRONG_TASKS`, use intent to decide. Keep the `task_complexity` parameter for backward compat.
**Better approach:** Add a separate method like `route_by_intent(intent: IntentClassification | None, fallback: TaskComplexity) -> ChatProvider` to keep concerns separate.

### Pitfall 4: Escalation Chain vs Existing Retry Logic
**What goes wrong:** The escalation chain conflicts with the existing `invalid_json` retry counter
**Why it happens:** The existing code at graph.py:1256-1280 already increments `invalid_json` and applies max retries. The escalation chain is a parallel concept.
**How to avoid:** The escalation chain should be WITHIN the existing retry flow. Use `consecutive_format_drift` as a sub-counter that determines which escalation step to use, while `invalid_json` remains the overall retry budget. The chain kicks in specifically when `validate_action()` normalizes successfully (used_fallback=True) but the raw output was non-canonical.

### Pitfall 5: Few-Shot Examples Too Large for Context
**What goes wrong:** Adding 2-3 JSON examples per role (each ~100-200 tokens) might push compact tier over budget
**Why it happens:** 16GB hardware allows more context than originally assumed, but phi4 at 8192 tokens is still tight with tool descriptions + few-shot
**How to avoid:** On compact tier, either skip few-shot entirely or include exactly 1 minimal example. On full tier, include all 2-3 examples. This aligns with the existing tier philosophy.

### Pitfall 6: args_schema Requires Updating All 36 Tools
**What goes wrong:** Partial migration leaves some tools with empty schemas
**Why it happens:** 36 tool files need `_args_schema` class variables added
**How to avoid:** Use the fallback to `required_args()` in the base class `args_schema` property. Migrate tools incrementally -- start with the most-used tools (write_file, sort_array, math_stats, string_ops, text_analysis, data_analysis, run_bash, read_file, read_file_chunk, outline_code) and let the rest fall back. Or do all 36 in one plan since the pattern is mechanical.

### Deterministic Fallback Classification
```python
def _deterministic_classify(plan: StructuredPlan) -> IntentClassification:
    top_level = [s for s in plan.steps if s.parent_id is None]
    all_tools = set()
    for s in plan.steps:
        all_tools.update(s.suggested_tools)

    complex_tools = {"data_analysis", "outline_code", "read_file_chunk", "run_bash", "http_request"}
    has_complex = bool(all_tools & complex_tools)
    multi_step = len(top_level) >= 3
    regex_fallback = plan.parsing_method == "regex_fallback"

    if has_complex or multi_step or regex_fallback:
        return IntentClassification(
            complexity="complex", mission_type="multi_step",
            confidence=0.6, source="deterministic_fallback"
        )
    return IntentClassification(
        complexity="simple", mission_type="tool_call",
        confidence=0.7, source="deterministic_fallback"
    )
```

## Discretion Recommendations

### Classify-only vs classify+decompose
**Recommendation: Classify-only.** The deterministic parser already decomposes input into steps. Adding LLM decomposition creates a second source of truth for plan structure, which conflicts with the existing `_enforce_step_limit()` and dependency inference. The classifier's value is routing (simple/complex), not restructuring the plan. Decomposition can be added in a future phase if deterministic parsing proves insufficient.

### Token estimation method
**Recommendation: `len//4`.** The project already uses this at graph.py:1058. Adding tiktoken means a new dependency, model-specific tokenizers, and inconsistency with existing estimation. The 4:1 ratio is close enough for budget enforcement (within ~20% of actual for English text).

### Truncation priority order
**Recommendation: Tool descriptions first, then few-shot examples.** Rationale: (1) tool descriptions are auto-generated and can be regenerated at any time, (2) few-shot examples are hand-crafted teaching material with higher value density, (3) on compact tier, tool descriptions are already minimal (just names + args), so truncating further means dropping tools from the list -- which is more harmful than dropping a few-shot example.

### FEW_SHOT tier injection
**Recommendation: Full tier only.** The compact tier exists because models have 8192 token context windows. Adding few-shot examples (each ~100-150 tokens, 2-3 per role = 200-450 tokens) consumes 5-8% of the window. On full tier (32K+), this is negligible. On compact tier, the system prompt + tool names already consume a significant fraction. Keep compact tier lean. If the user wants to test few-shot on compact, the code path should be easy to enable later.

### Compact tier threshold
**Recommendation: Keep at 10k.** The threshold is well-tested and documented. Changing it would affect existing behavior and test expectations without clear benefit. The 16GB hardware note is about total model memory, not context window -- phi4 at 8192 tokens is still below 10k regardless of RAM.

## Open Questions

1. **How many tools should have args_schema in Plan 3 vs deferred?**
   - What we know: 36 tools exist. Most have simple args (1-3 params).
   - What's unclear: Whether all 36 need migration in this phase or if incremental is acceptable.
   - Recommendation: Migrate all 36 in one pass. Each migration is 3-5 lines of class variable. Mechanical, low-risk, and eliminates the fragile regex path entirely.

2. **Should the classifier use response_schema for guaranteed JSON?**
   - What we know: `ChatProvider.generate()` already accepts `response_schema`. LlamaCpp uses it for GBNF grammar.
   - What's unclear: Whether 500ms is enough for grammar-constrained generation on small models.
   - Recommendation: Use response_schema when available (it produces valid JSON more reliably), but the 500ms timeout catches the case where grammar makes generation too slow. The deterministic fallback is always there.

3. **Where exactly does the escalation chain hook into _plan_next_action()?**
   - What we know: The current exception handler at graph.py:1250 catches ValueError from validate_action.
   - What's unclear: The escalation chain fires on "parseable but non-canonical" output (used_fallback=True), NOT on total parse failures.
   - Recommendation: Check `used_fallback` from `validate_action()` return. When True AND action is valid, that's "drift" (model is consistently non-canonical). When validation raises ValueError, that's a hard failure (existing retry logic applies). The two counters track different failure modes.
