# Phase 07.7: Hybrid Intent Classifier + Few-Shot Prompt Refinement - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Layer an LLM-based intent classifier on top of the deterministic `mission_parser.py` to classify mission type and complexity before planning begins. Add 2-3 concrete few-shot JSON examples to each directive (supervisor, executor, evaluator). Add COMPACT sections to executor.md and evaluator.md. Replace fragile regex `Tool.required_args()` with typed `Tool.args_schema`. Enforce per-role token budgets with tool description truncation. Wire intent classification into ModelRouter to replace explicit complexity params. Implement format correction escalation chain (hint → retry → accept).

</domain>

<decisions>
## Implementation Decisions

### Intent Classifier Design
- Lives inside `parse_missions()` with optional `ChatProvider` parameter
- After deterministic parsing produces StructuredPlan, LLM classifies intent/complexity in ONE quick call (~300 tokens)
- Binary classification: `simple` → fast provider, `complex` → strong provider
- 500ms timeout with deterministic fallback (infer from step count + keywords)
- Result stored in `StructuredPlan.intent_classification: dict | None`
- Key value: real user input is natural language (not numbered tasks), so deterministic parser always falls back to regex — the classifier adds actual understanding of task complexity
- Claude's discretion: whether to classify-only or also decompose natural language into sub-steps

### Few-Shot Examples
- 2-3 JSON examples per role (supervisor, executor, evaluator)
- Include one store-then-query pattern example in supervisor few-shot: `outline_code` → `read_file_chunk` loop → `write_file` partial output
- Placed in `## FEW_SHOT` section in each directive
- COMPACT sections added to executor.md and evaluator.md (matching supervisor.md pattern from 7.6)
- Claude's discretion: tier injection strategy for FEW_SHOT (full-only or both tiers with fewer on compact) — 16GB hardware allows more context than assumed, and executor models are smaller so relative context room is larger

### Token Budget & Prompt Sizing
- Per-role token budgets enforced: classifier ~300, planner ~1000, executor ~300
- `_estimate_prompt_tokens()` helper added — Claude's discretion on method (len//4 or tiktoken)
- Truncation priority when over budget — Claude's discretion (tool descriptions first vs few-shot first)
- **Typed Tool.args_schema replaces fragile regex `required_args()`** — each tool declares its args as a dict or typed structure, eliminating regex parsing failures from 7.6
- `_build_action_json_schema()` and compact prompt signatures consume `args_schema` instead of regex output

### Format Correction Escalation Chain
- **Step 1 (first drift):** Inject `[Orchestrator]` user-message hint with correct format example (free, no LLM call)
- **Step 2 (second consecutive drift):** Retry with explicit system-level correction (costs 1 LLM call)
- **Step 3 (third consecutive drift):** Accept normalized output from `validate_action()` + log WARNING. No more retries — model consistently produces non-canonical but parseable output
- Consecutive drift counter resets on a correctly-formatted response
- **New structural_health counters:** `format_correction_hints` (step 1 events) and `format_retries` (step 2 events) — visible in audit_report

### Intent-Driven ModelRouter Wiring
- `StructuredPlan.intent_classification` complexity feeds into `ModelRouter.route()`
- `"complex"` → strong provider, `"simple"` → fast provider
- Replaces the current pattern where callers always pass explicit `TaskComplexity` labels (today always `"planning"` → always strong)
- Backward compatible: if intent_classification is None (classifier skipped/failed), falls back to existing explicit routing

### Claude's Discretion
- Classify-only vs classify+decompose for natural language input
- Token estimation method (len//4 vs tiktoken)
- Truncation priority order (tool descriptions vs few-shot examples)
- FEW_SHOT tier injection (full-only or both tiers given 16GB hardware)
- Compact tier threshold: keep at 10k or adjust given hardware constraints

</decisions>

<specifics>
## Specific Ideas

- Real user input is natural language, not numbered/bulleted tasks — parser always falls back to regex, making the classifier especially valuable for routing decisions
- phi4 consistently produces `{"tool": "X", flat_args}` instead of canonical format — the escalation chain addresses this directly
- 16GB hardware means llama-cpp context window can go beyond 8192 (phi4 supports up to 16k) — FEW_SHOT examples may fit even on compact tier
- Executor models are smaller, so relative context budget for few-shot is larger on those roles
- Store-then-query pattern: 7.7 teaches via few-shot examples, 7.9 implements the `query_context` tool

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `mission_parser.py`: `parse_missions()` entry point, `StructuredPlan` dataclass, `_TOOL_KEYWORD_MAP` (200+ keyword-to-tool mappings)
- `model_router.py`: `ModelRouter` with `route(task_complexity)`, `TaskComplexity` literal, `_STRONG_TASKS` frozenset, `has_dual_providers` property
- `validate_action()` in `action_parser.py`: already normalizes `{"tool": "X", flat_args}` → canonical format — escalation chain builds on top
- `structural_health` dict in RunState: existing `json_parse_fallback`, `schema_mismatch` counters — add new counters here
- `_build_system_prompt()` in graph.py: compact/full tier builder, reads `## COMPACT` from supervisor.md
- `Tool.required_args()` in `tools/base.py`: regex-based arg extraction — being replaced by typed `args_schema`

### Established Patterns
- `[Orchestrator]` prefix for injected user-role messages (context hints, corrections, eviction summaries)
- `TYPE_CHECKING` import guard for optional dependencies
- `.model_dump()` at RunState boundaries for Pydantic models
- `pool=None` no-op guards for Postgres-optional code paths
- Compact tier reads `## SECTION_NAME` from directive files at build time

### Integration Points
- `parse_missions()` in mission_parser.py: add optional `ChatProvider` param for LLM classifier
- `_build_system_prompt()` in graph.py: inject `## FEW_SHOT` sections, enforce per-role budgets
- `ModelRouter.route()`: accept intent classification instead of explicit TaskComplexity
- `_plan_next_action()` in graph.py: implement escalation chain (drift detection, hint injection, retry)
- `structural_health` dict: add `format_correction_hints` and `format_retries` counters
- `tools/base.py`: add `args_schema` property/method to Tool base class

</code_context>
