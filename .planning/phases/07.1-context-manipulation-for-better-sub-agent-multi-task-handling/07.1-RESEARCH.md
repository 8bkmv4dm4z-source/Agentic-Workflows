# Phase 7.1: Context Manipulation for Better Sub-Agent Multi-Task Handling - Research

**Researched:** 2026-03-07
**Domain:** LangGraph state management, message context window optimization, Pydantic modeling
**Confidence:** HIGH

## Summary

This phase redesigns how the orchestrator manages context (message history, tool results, mission state) during multi-mission runs. The current system has three separate context manipulation mechanisms -- `_evict_tool_result_messages()` (Ollama-specific), `_compact_messages()` (sliding window), and `_mission_handoff_hint()` (ad-hoc mission transition injection) -- that operate independently without a unified model. The new design introduces a `MissionContext` Pydantic model as a durable queryable store, replaces the fragmented eviction logic with an event-driven provider-agnostic system, and enriches specialist subgraph context with mission goal and prior results.

The codebase is well-structured for this change. RunState is a TypedDict with reducer-aware list fields. Specialist subgraphs (ExecutorState, EvaluatorState) are isolated TypedDicts with prefixed fields. The existing `_emit_trace()` pattern provides observability hooks. Tool results are deterministic JSON structures amenable to programmatic summary extraction.

**Primary recommendation:** Build MissionContext as a Pydantic BaseModel stored in `mission_contexts: dict[str, MissionContext]` on RunState (string keys for JSON serialization safety). Replace all three existing context mechanisms with a single ContextManager class that handles summarization, eviction, and specialist enrichment through one abstraction.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- On mission completion, summarize all of Mission N's tool result messages into a compact summary and replace them in the planner's message history
- Summary includes: mission goal, tools used, key results, and cross-mission artifacts (file paths, computed values, URLs discovered during execution that later missions may need)

- New top-level fields on ExecutorState and EvaluatorState TypedDicts (not crammed into input_context dict)
- Prior results summary uses the same format as mission boundary summaries -- one abstraction, tested once
- Replace the existing Ollama-specific `_evict_tool_result_messages()` entirely with a provider-agnostic eviction system

- Full observability: every eviction event emits both a log entry and a Langfuse trace event
- Typed Pydantic model (`MissionContext`) stored as `dict[int, MissionContext]` on RunState
- Queryable context store pattern: planner queries MissionContext for what it needs
- Hierarchical sub-mission support: MissionContext supports nested sub-mission contexts (sub_mission_id level), matching structured plan architecture
- Context is checkpointed automatically via RunState (no new external store)
- Isolated + explicit artifacts for cross-mission data flow

### Pattern 1: MissionContext as Pydantic BaseModel
**What:** A structured Pydantic model representing per-mission context with hierarchical sub-mission support
**When to use:** All mission context operations -- creation, update, querying, summarization
**Example:**
```python
from pydantic import BaseModel, Field

class ArtifactRecord(BaseModel):
    """A cross-mission artifact extracted from tool results."""
    key: str                    # e.g., "file_path", "computed_value"
    value: str                  # the actual artifact value
    source_tool: str            # which tool produced it
    source_mission_id: int      # which mission produced it

class SubMissionContext(BaseModel):
    """Context for a sub-mission step."""
    sub_mission_id: str         # matches MissionStep.id (e.g., "1a", "1.1")
    goal: str
    status: str = "pending"
    tools_used: list[str] = Field(default_factory=list)
    key_results: dict[str, str] = Field(default_factory=dict)

class MissionContext(BaseModel):
    """Durable context for a single mission."""
    mission_id: int
    goal: str
    status: str = "pending"
    tools_used: list[str] = Field(default_factory=list)
    key_results: dict[str, str] = Field(default_factory=dict)
    artifacts: list[ArtifactRecord] = Field(default_factory=list)
    sub_missions: dict[str, SubMissionContext] = Field(default_factory=dict)
    summary: str = ""           # populated on mission completion
    message_indices: list[int] = Field(default_factory=list)  # track which messages belong to this mission

    def build_summary(self) -> str:
        """Build deterministic summary from structured data."""
        parts = [f"Mission {self.mission_id}: {self.goal}"]
        if self.tools_used:
            parts.append(f"Tools: {' -> '.join(self.tools_used)}")
        for k, v in self.key_results.items():
            parts.append(f"{k}: {v[:200]}")
        if self.artifacts:
            artifact_strs = [f"{a.key}={a.value}" for a in self.artifacts]
            parts.append(f"Artifacts: {', '.join(artifact_strs)}")
        return " | ".join(parts)
```

### Anti-Patterns to Avoid
- **Storing MissionContext in a separate external store:** The user explicitly decided it checkpoints via RunState. No new store.
- **Using LLM for summarization:** All summaries must be deterministic. Extract fields from structured JSON.
- **Cramming enrichment into `input_context` dict:** User explicitly decided on new top-level fields on specialist TypedDicts.
- **Keeping the old `_evict_tool_result_messages()`:** User decided to replace it entirely, not layer on top.
- **Using `role="system"` for injected summaries:** Must use `role="user"` with `[Orchestrator]` prefix (Phase 5 decision).

## Common Pitfalls

### Pitfall 1: RunState TypedDict dict reducer incompatibility
**What goes wrong:** `dict[str, MissionContext]` on RunState needs careful handling since TypedDict dict fields use replacement semantics (not merge). Parallel Send() branches would overwrite each other.
**Why it happens:** List fields use `Annotated[list[T], operator.add]` reducers for append safety. Dict fields have no built-in merge reducer.
**How to avoid:** Since parallel Send() is deferred (out of scope), dict replacement is safe for now. Document that when PRLL-01 is implemented, `mission_contexts` will need a custom reducer that merges by key. For now, use a plain `dict[str, Any]` field (serialized MissionContext dicts).
**Warning signs:** If you see `mission_contexts` losing entries, a concurrent write happened without a merge reducer.

### Pitfall 2: Pydantic model in TypedDict serialization
**What goes wrong:** LangGraph checkpointers serialize state as JSON. Pydantic BaseModel objects in a TypedDict field may not serialize cleanly through all checkpointer backends.
**Why it happens:** TypedDict expects plain dicts; BaseModel instances need `.model_dump()`.
**How to avoid:** Store `mission_contexts` as `dict[str, dict[str, Any]]` on RunState (serialized form). Use `MissionContext.model_validate(data)` when reading, `.model_dump()` when writing. The Pydantic model is a domain object, not a state field type.
**Warning signs:** Checkpointer save/load failures, JSON serialization errors.

### Pitfall 3: Message index tracking drift
**What goes wrong:** Tracking which message indices belong to which mission breaks when messages are inserted, deleted, or reordered by compaction.
**Why it happens:** Other code paths (progress hints, handoff hints, compaction) mutate `state["messages"]` independently.
**How to avoid:** Don't track by index. Instead, tag messages with a mission_id marker in the content (e.g., `[M{id}]` prefix) or use content-based identification (messages starting with `TOOL RESULT` between mission boundaries). Alternatively, track message count at mission start/end and use offsets.
**Warning signs:** Summary replaces wrong messages, or misses messages belonging to a mission.

### Pitfall 4: Consecutive system messages breaking Ollama
**What goes wrong:** Ollama JSON mode fails when consecutive `role="system"` messages appear in the transcript.
**Why it happens:** Phase 5 discovered this; the fix was to use `role="user"` with `[Orchestrator]` prefix.
**How to avoid:** All injected context (summaries, progress hints, handoff context) MUST use `role="user"` with `[Orchestrator]` prefix. Never inject `role="system"` except for the initial system prompt.
**Warning signs:** Ollama returning empty responses or JSON parse errors after context injection.

### Pitfall 5: ensure_state_defaults compaction conflict
**What goes wrong:** `ensure_state_defaults()` in state_schema.py already has a sliding window compaction (line 258-264) that fires independently of the new ContextManager eviction.
**Why it happens:** The function runs on every node entry and blindly drops oldest non-system messages over a threshold.
**How to avoid:** Either remove the compaction from `ensure_state_defaults()` (since ContextManager replaces it) or coordinate thresholds so they don't conflict. Removing it is cleaner -- the ContextManager should be the single source of truth for message lifecycle.
**Warning signs:** Messages disappearing unexpectedly, mission summaries being dropped by the state_defaults compaction before the ContextManager processes them.

### Pitfall 6: _compact_messages and new ContextManager double-eviction
**What goes wrong:** `_compact_messages()` in graph.py (line 2300) calls `_evict_tool_result_messages()` and then does its own sliding window. If ContextManager also does eviction, messages get double-processed.
**Why it happens:** Three separate compaction/eviction mechanisms exist today.
**How to avoid:** Replace `_compact_messages()` entirely with ContextManager calls. The ContextManager subsumes all three functions: `_evict_tool_result_messages()`, `_compact_messages()`, and the `ensure_state_defaults()` compaction.
**Warning signs:** Message count dropping faster than expected, context summaries being deleted.

### Deterministic summary extraction from tool result
```python
def extract_summary_from_result(tool_name: str, result: dict[str, Any]) -> dict[str, str]:
    """Extract key fields from a tool result for summary inclusion."""
    if "error" in result:
        return {"error": str(result["error"])[:200]}

    summary: dict[str, str] = {}

    if tool_name == "write_file":
        # result: {"result": "Successfully wrote N characters to path"}
        summary["outcome"] = str(result.get("result", ""))[:200]
    elif tool_name == "sort_array":
        sorted_val = result.get("sorted", result.get("result", []))
        summary["sorted_count"] = str(len(sorted_val)) if isinstance(sorted_val, list) else "unknown"
    elif tool_name == "data_analysis":
        for key in ("mean", "median", "non_outliers", "outliers"):
            if key in result:
                val = result[key]
                summary[key] = str(val)[:100] if not isinstance(val, list) else f"[{len(val)} items]"
    elif tool_name == "math_stats":
        for key in ("sum", "mean", "median"):
            if key in result:
                summary[key] = str(result[key])
    else:
        # Generic: first 200 chars of stringified result
        summary["result"] = str(result)[:200]

    return summary
```

### Eviction event with observability
```python
def _emit_eviction_event(
    self,
    state: RunState,
    *,
    trigger: str,       # "mission_complete" | "large_result" | "step_threshold" | "sliding_window"
    mission_id: int,
    messages_removed: int,
    summary_injected: bool,
) -> None:
    """Emit eviction event to both logger and Langfuse trace."""
    logger.info(
        "CONTEXT EVICT trigger=%s mission_id=%s messages_removed=%s summary=%s",
        trigger, mission_id, messages_removed, summary_injected,
    )
    # Reuse existing _emit_trace pattern
    policy_flags = state.get("policy_flags", {})
    trace = policy_flags.setdefault("pipeline_trace", [])
    trace.append({
        "stage": "context_eviction",
        "step": state.get("step", 0),
        "trigger": trigger,
        "mission_id": mission_id,
        "messages_removed": messages_removed,
        "summary_injected": summary_injected,
    })
```

## Open Questions

1. **Message tagging for per-mission attribution**
   - What we know: Messages are plain dicts with role/content. No mission_id tag exists.
   - What's unclear: Best way to know which messages belong to which mission for eviction.
   - Recommendation: Track mission boundaries by step number in MissionContext (record step range), not by tagging individual messages. Simpler, no existing message format changes needed.

2. **Interaction with token_budget tracking**
   - What we know: `token_budget_remaining` and `token_budget_used` are tracked via len//4 estimation. Eviction reduces message size but these counters don't get updated.
   - What's unclear: Whether eviction should update the token budget counters.
   - Recommendation: Do not update counters on eviction -- they track total usage across the run, not current context size. The eviction is about context window management, not budget accounting.

3. **Threshold defaults**
   - What we know: User said sliding window ~30 messages, large result threshold unspecified.
   - What's unclear: Optimal thresholds for different providers.
   - Recommendation: Make configurable via env vars with sensible defaults: `CTX_SLIDING_WINDOW=30`, `CTX_LARGE_RESULT_CHARS=4000`, `CTX_STEP_THRESHOLD=10`. Can be tuned per-provider without code changes.
