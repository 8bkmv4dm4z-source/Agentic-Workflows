# Phase 07.9: Dynamic Context Querying + Memory Consolidation + Compliance Observability - Research

**Researched:** 2026-03-10
**Domain:** Tool implementation, semantic clustering, Langfuse custom metrics, cross-run analytics
**Confidence:** HIGH

## Summary

Phase 07.9 adds three independent capabilities to the existing platform: (1) a `query_context` tool that exposes `MissionContextStore.query_cascade()` to the LLM during planning, (2) a memory consolidation module that clusters old mission contexts by cosine similarity and replaces clusters with summaries, and (3) schema compliance observability via Langfuse custom scores and cross-run dashboard in `run_audit.py`.

All three features build directly on existing infrastructure. `query_context` mirrors the existing `retrieve_run_context` tool pattern and plugs into `MissionContextStore` which already has the full 5-layer cascade. Memory consolidation uses the existing `mission_contexts` Postgres table and embedding infrastructure. Compliance reporting uses Langfuse 3.14.5's `create_score()` API and the existing `structural_health` counters already tracked in `RunState`.

**Primary recommendation:** Implement as 5 sequential plans matching the success criteria. No new dependencies required -- numpy is not needed because cosine similarity on 384-dim vectors can be computed with pure Python (dot product / magnitude), and the consolidation operates on small result sets from Postgres queries.

### Pattern 2: Tool with Store Injection
**What:** Tool subclass receives store in __init__, delegates to store methods
**When to use:** query_context tool
**Example:**
```python
# Follows retrieve_run_context.py pattern exactly
class QueryContextTool(Tool):
    name = "query_context"
    _args_schema = {
        "query": {"type": "string", "required": "true"},
        "max_results": {"type": "number"},
    }
    description = (
        "Query cross-run mission context for similar past missions. "
        "Required args: query (str). Optional: max_results (int, default 3)."
    )

    def __init__(self, store: MissionContextStore) -> None:
        self.store = store

    def execute(self, args: dict[str, Any]) -> dict[str, Any]:
        query = str(args.get("query", "")).strip()
        if not query:
            return {"error": "query is required"}
        max_results = min(int(args.get("max_results", 3)), 10)
        results = self.store.query_cascade(goal=query, top_k=max_results)
        if not results:
            return {"results": [], "count": 0}
        return {
            "results": [
                {"goal": r["goal"], "summary": r["summary"], "tools_used": r["tools_used"], "score": r["score"]}
                for r in results
            ],
            "count": len(results),
        }
```

### Pattern 3: Langfuse Score Reporting with Graceful Degradation
**What:** Report numeric scores to Langfuse; no-op when not configured
**When to use:** After each parse attempt in graph.py
**Example:**
```python
# observability.py addition
def report_schema_compliance(
    role: str,
    first_attempt_success: bool,
    trace_id: str | None = None,
) -> None:
    """Report schema compliance as a Langfuse score. No-op if not configured."""
    client = get_langfuse_client()
    if client is None:
        return
    try:
        client.create_score(
            name="schema_compliance",
            value=1.0 if first_attempt_success else 0.0,
            data_type="NUMERIC",
            trace_id=trace_id,
            comment=f"role={role}",
        )
    except Exception:  # noqa: BLE001
        pass  # graceful degradation
```

### Pattern 4: Pure Python Cosine Similarity
**What:** Compute cosine similarity without numpy
**When to use:** Memory consolidation clustering
**Example:**
```python
def _cosine_similarity(a: list[float], b: list[float]) -> float:
    dot = sum(x * y for x, y in zip(a, b))
    mag_a = sum(x * x for x in a) ** 0.5
    mag_b = sum(x * x for x in b) ** 0.5
    if mag_a == 0 or mag_b == 0:
        return 0.0
    return dot / (mag_a * mag_b)
```

### Pattern 5: Greedy Single-Linkage Clustering
**What:** Cluster items by similarity threshold without scipy/sklearn
**When to use:** consolidate_memories() grouping
**Example:**
```python
def _cluster_by_similarity(
    items: list[dict],
    threshold: float = 0.85,
) -> list[list[dict]]:
    """Greedy single-linkage clustering by cosine similarity."""
    clusters: list[list[dict]] = []
    used: set[int] = set()
    for i, item_a in enumerate(items):
        if i in used:
            continue
        cluster = [item_a]
        used.add(i)
        for j, item_b in enumerate(items):
            if j in used:
                continue
            if _cosine_similarity(item_a["embedding"], item_b["embedding"]) >= threshold:
                cluster.append(item_b)
                used.add(j)
        clusters.append(cluster)
    return clusters
```

### Anti-Patterns to Avoid
- **Do not add numpy as a dependency** for cosine similarity on 384-dim vectors. The consolidation batch size is small (missions older than 7 days from one DB query). Pure Python is sufficient and avoids dependency bloat.
- **Do not call Langfuse synchronously in the hot path** without a timeout. The `create_score()` call should be fire-and-forget (already async internally in langfuse SDK via background thread).
- **Do not modify `_TOOL_KEYWORD_MAP` entries for "previous" without checking existing mappings.** Currently "previous" maps to `retrieve_run_context`. SC-2 requires adding `query_context` to that list (append, don't replace).
- **Do not break the TOOL_BITS append-only policy.** `query_context` must be appended at bit position 37 (next available after `retrieve_run_context` at 36).

## Common Pitfalls

### Pitfall 1: Keyword Map Collision for "previous"
**What goes wrong:** "previous" already maps to `retrieve_run_context` in `_TOOL_KEYWORD_MAP`. Overwriting it breaks the existing mapping.
**Why it happens:** SC-2 says map "previous" to query_context, but it's already mapped.
**How to avoid:** Append `query_context` to the existing list: `"previous": ["retrieve_run_context", "query_context"]`. Same for "context" and "history" if desired.
**Warning signs:** Tests for retrieve_run_context keyword mapping fail.

### Pitfall 2: query_context Tool Without Embedding Provider
**What goes wrong:** `query_cascade()` needs an embedding for the L4 (HNSW cosine) layer. If the tool only passes `goal` without embedding, L4 is skipped (returns fewer results).
**Why it happens:** The tool has no direct access to EmbeddingProvider.
**How to avoid:** Pass the EmbeddingProvider to QueryContextTool alongside MissionContextStore. The tool calls `provider.embed([query])` to get the embedding, then passes it to `query_cascade(goal=query, embedding=embedding)`. When provider is None, pass embedding=None (L2 BM25 still works).
**Warning signs:** query_context results never come from L4/RRF layers.

### Pitfall 3: Consolidation Deleting Active Run Missions
**What goes wrong:** If consolidation runs while a mission is in progress, it could delete contexts being used.
**Why it happens:** The >7 days filter may not be sufficient if old runs are resumed.
**How to avoid:** Add a `status = 'completed'` filter to the consolidation query. Only consolidate completed missions. The transactional boundary (single connection) ensures atomicity.
**Warning signs:** Cross-run injection returns stale/missing results after consolidation.

### Pitfall 4: run_audit.py Only Reads SQLite
**What goes wrong:** `run_audit.py` currently only reads from SQLite checkpoint DB. The compliance dashboard needs structural_health data which is in the state JSON.
**Why it happens:** run_audit.py uses `_load_latest_states()` which reads `state_json` column. `structural_health` is already in state.
**How to avoid:** structural_health is already accessible via `state.get("structural_health", {})` in the summarize loop. Just add the compliance columns.
**Warning signs:** None -- this is straightforward.

### Pitfall 5: Langfuse trace_id Not Available in graph.py Parse Path
**What goes wrong:** `report_schema_compliance()` needs a trace_id but graph.py may not have an active Langfuse trace.
**Why it happens:** Langfuse callbacks are wired via ContextVar, not directly accessible.
**How to avoid:** Make trace_id optional in `report_schema_compliance()`. When None, Langfuse SDK can create a standalone score (scores can exist without a trace_id if using session_id or standalone). Alternatively, use `run_id` as a session-level identifier.
**Warning signs:** Scores created without trace association are harder to query in Langfuse UI.

### Pitfall 6: TOOL_BITS Bit Position for query_context
**What goes wrong:** If query_context is assigned an incorrect bit position, bitmask queries are corrupted.
**Why it happens:** TOOL_BITS is append-only at mission_context_store.py:32-71. Current max is bit 36 (retrieve_run_context).
**How to avoid:** Assign `"query_context": 37` -- the next available position. Update TOOL_BITS in mission_context_store.py.
**Warning signs:** Bitmask-based cascade queries return wrong results.

### consolidate_memories() Skeleton
```python
# storage/memory_consolidation.py
def consolidate_memories(
    pool: ConnectionPool,
    embedding_provider: EmbeddingProvider | None = None,
    age_days: int = 7,
    similarity_threshold: float = 0.85,
) -> dict[str, Any]:
    """Cluster old missions by semantic similarity and replace with summaries."""
    with pool.connection() as conn:
        # 1. Fetch old completed missions with embeddings
        rows = conn.execute(
            "SELECT id, run_id, mission_id, goal, summary, tools_used, embedding "
            "FROM mission_contexts "
            "WHERE status = 'completed' AND created_at < NOW() - INTERVAL '%s days'",
            (age_days,),
        ).fetchall()
        if not rows:
            return {"clusters": 0, "consolidated": 0, "kept": 0}

        # 2. Build items with embeddings
        items = [_row_to_item(row) for row in rows]

        # 3. Cluster by cosine similarity
        clusters = _cluster_by_similarity(items, threshold=similarity_threshold)

        # 4. For each cluster with 2+ items, create summary and replace
        consolidated = 0
        for cluster in clusters:
            if len(cluster) < 2:
                continue
            summary = _merge_cluster_summary(cluster)
            ids_to_delete = [item["id"] for item in cluster]
            # Transactional: delete originals, insert summary
            conn.execute(
                "DELETE FROM mission_contexts WHERE id = ANY(%s)",
                (ids_to_delete,),
            )
            # Insert consolidated record (use first item as base)
            _insert_consolidated(conn, cluster[0], summary, embedding_provider)
            consolidated += len(cluster)

    return {"clusters": len(clusters), "consolidated": consolidated, "kept": len(rows) - consolidated}
```

### report_schema_compliance() in observability.py
```python
# Source: Langfuse 3.x create_score() API
def report_schema_compliance(
    role: str,
    first_attempt_success: bool,
    trace_id: str | None = None,
    run_id: str | None = None,
) -> None:
    """Report schema compliance to Langfuse as a numeric score (1.0 or 0.0)."""
    client = get_langfuse_client()
    if client is None:
        return
    import contextlib
    with contextlib.suppress(Exception):
        kwargs: dict[str, Any] = {
            "name": "schema_compliance",
            "value": 1.0 if first_attempt_success else 0.0,
            "data_type": "NUMERIC",
            "comment": f"role={role}",
        }
        if trace_id:
            kwargs["trace_id"] = trace_id
        if run_id:
            kwargs["session_id"] = run_id
        client.create_score(**kwargs)
```

## Open Questions

1. **Consolidation embedding for summary records**
   - What we know: After merging a cluster into a summary, we need an embedding for the new record so it participates in future cascade queries.
   - What's unclear: Should we re-embed the merged summary text, or average the cluster's embeddings?
   - Recommendation: Average the cluster embeddings (faster, no embedding call needed). If EmbeddingProvider is available, re-embed the summary text for better quality. Fall back to averaging when provider is None.

2. **Langfuse trace association**
   - What we know: `create_score()` works with or without trace_id. Graph.py may not have easy access to the current Langfuse trace ID.
   - What's unclear: Whether the Langfuse callback handler exposes the active trace ID.
   - Recommendation: Use `run_id` as session_id for score grouping. If trace_id is obtainable from the callback handler, pass it. Otherwise session-level grouping is sufficient for compliance dashboards.

3. **CLI entry point for --consolidate**
   - What we know: run.py already has argparse. run_audit.py is the cross-run CLI.
   - What's unclear: Should --consolidate go on run.py or run_audit.py?
   - Recommendation: Add to run_audit.py since it's the maintenance/analytics CLI. run.py is for running agents.
