From src/agentic_workflows/orchestration/langgraph/context_manager.py (attribution computation):
```python
# Lines ~621-625 of build_planner_context_injection():
attribution_parts = [
    f"{hit.get('source_layer', '?')}:{hit.get('score', 0.0):.2f}"
    for hit in hits
]
attribution = ", ".join(attribution_parts) if attribution_parts else "-"
```
With source_layer="L0" and score=0.9, attribution will be "L0:0.90" instead of "?:0.90".

The attribution string is emitted in the CONTEXT INJECT log line. It is NOT part of the string returned by build_planner_context_injection() directly — the attribution is only logged. Therefore, the test assertion should check the log output OR check the returned string for the presence/absence of a pattern.

IMPORTANT: The returned string from build_planner_context_injection() does NOT include the attribution — it includes the cross-run content. The attribution only appears in the log. To verify attribution in a unit test without log capture, the simplest approach is:
1. Verify that [Cross-run] Similar: appears in the result (confirming the hit was processed)
2. Add a test that directly calls the attribution computation logic via a mock, OR
3. Use `caplog` pytest fixture to capture log output and assert "L0:0.90" appears in the CONTEXT INJECT log line

Option 3 (caplog) is the most direct. The test method signature becomes:
```python
def test_attribution_shows_real_layer(self, caplog):
    import logging
    with caplog.at_level(logging.INFO, logger="context_manager"):
        result = cm.build_planner_context_injection(state)
    assert "L0:0.90" in caplog.text  # not "?:0.90"
    assert "?:0.90" not in caplog.text
```
