# Phase 07.5: Wire ArtifactStore to Runtime — Research

**Researched:** 2026-03-09
**Domain:** Postgres artifact persistence, ContextManager wiring, LangGraphOrchestrator parameter forwarding
**Confidence:** HIGH

## Summary

`ArtifactStore` exists and is fully implemented (`src/agentic_workflows/storage/artifact_store.py`). The
`mission_artifacts` table already exists via migration `004_mission_artifacts.sql`. The problem is entirely
a wiring gap: `ArtifactStore` is instantiated in `app.py` (stored in `application.state.artifact_store`)
but never passed to `LangGraphOrchestrator` or `ContextManager`, and `_persist_mission_context()` never
calls `artifact_store.upsert()`.

The `MissionContext.artifacts` list is already populated correctly by `on_tool_result()` before
`on_mission_complete()` is called. At the point `on_mission_complete()` builds `persist_ctx` (line 468 in
`context_manager.py`), the `ctx.artifacts` list is available on the reconstructed `MissionContext` object
— it just is not iterated and persisted. The fix is purely additive: no schema changes, no migration work,
no changes to the artifact extraction pipeline.

There are two parallel wiring chains: the `ContextManager` must receive an `artifact_store` param, and
`LangGraphOrchestrator` must accept and forward it. Both `run.py` and `user_run.py` already contain a
`_build_orchestrator()` / `__post_init__` block that conditionally creates the `pg_pool` and
`MissionContextStore` — `ArtifactStore` needs to be created there as well and passed through. The
`app.py` lifespan already creates `ArtifactStore` but does NOT forward it to the orchestrator; that gap
must also be closed.

**Primary recommendation:** Add `artifact_store: ArtifactStore | None = None` to `ContextManager.__init__`
and `LangGraphOrchestrator.__init__`, persist each `ArtifactRecord` in `_persist_mission_context()`, and
add a Postgres integration test that verifies `SELECT * FROM mission_artifacts WHERE run_id = ?` after a
mission completes.

## Architecture Patterns

### Pattern 1: Pool-injection parameter chain

The project follows a strict pool-injection pattern: stores never manage their own connection; a
`ConnectionPool` is created once at startup and injected via `__init__(pool=...)`. All three existing
stores (`PostgresCheckpointStore`, `PostgresMemoStore`, `MissionContextStore`) follow this pattern.
`ArtifactStore` already follows it too — `pool=None` is a graceful no-op.

**Wiring chain for this phase:**

```
pg_pool (created in run.py / user_run.py / app.py lifespan)
    → ArtifactStore(pool=pg_pool, embedding_provider=embedding_provider)
    → LangGraphOrchestrator(artifact_store=artifact_store, ...)
    → ContextManager(artifact_store=artifact_store, ...)
    → _persist_mission_context() calls artifact_store.upsert() per ArtifactRecord
```

### Pattern 2: TYPE_CHECKING import guard for optional store params

`context_manager.py` already uses `TYPE_CHECKING` for `MissionContextStore` and `EmbeddingProvider`
(lines 21–23). `ArtifactStore` must follow the same pattern — runtime import only in
`_persist_mission_context()` body or type-annotated as `ArtifactStore | None` under `TYPE_CHECKING`.

```python
# context_manager.py — add to TYPE_CHECKING block
if TYPE_CHECKING:
    from agentic_workflows.storage.artifact_store import ArtifactStore
```

At runtime the parameter is typed with a string annotation (`"ArtifactStore | None"`) or imported lazily.

### Pattern 3: on_mission_complete artifact persistence

At the point `on_mission_complete()` calls `_persist_mission_context()`, the `ctx` local variable is a
fully-reconstructed `MissionContext` with `.artifacts` populated. The `persist_ctx` dict currently omits
artifacts. Two approaches are valid:

**Approach A (preferred):** Persist artifacts directly inside `on_mission_complete()` after
`_persist_mission_context()`, iterating `ctx.artifacts` and calling `artifact_store.upsert()` for each.
This keeps `_persist_mission_context()` focused on the `MissionContextStore` upsert.

**Approach B:** Pass artifacts into `_persist_mission_context()` via the `persist_ctx` dict and handle
them there. This keeps artifact persistence co-located with mission context persistence.

The roadmap explicitly says "wire `_persist_mission_context()` to call `artifact_store.upsert()`", so
Approach B (or inline in `_persist_mission_context`) is what the planner expects. Either works since
the method is called from `on_mission_complete()` which holds the `ctx` object.

### Pattern 4: Embedding for artifact upsert

`ArtifactStore.upsert()` requires an `embedding: list[float]` argument. The project pattern for missing
embeddings is a zero vector:

```python
# ArtifactStore.upsert requires embedding — use zero vector when no provider
embedding = [0.0] * 384
if self._embedding_provider is not None:
    embedding = self._embedding_provider.embed_sync(artifact.value[:200])
```

`ContextManager` already holds `self._embedding_provider` — reuse it for artifact embeddings.

### Pattern 5: Conditional Postgres instantiation in run.py / user_run.py

Both entrypoints use the same conditional block gated on `DATABASE_URL`. The current block in `run.py`
`_build_orchestrator()` (lines 986–1012) already creates `pool`, `embedding_provider`, and
`mission_context_store`. Add `ArtifactStore` creation there:

```python
# In run.py _build_orchestrator(), inside the `if db_url:` block:
from agentic_workflows.storage.artifact_store import ArtifactStore
artifact_store = ArtifactStore(pool=pool, embedding_provider=embedding_provider)
```

Then pass `artifact_store=artifact_store` to `LangGraphOrchestrator(...)`.

Same pattern in `user_run.py` `UserSession.__post_init__()` (lines 72–102).

### Pattern 6: app.py already creates ArtifactStore — just forward it

`app.py` lifespan (lines 107–108) creates `artifact_store` but stores it only in
`application.state.artifact_store`. It does NOT pass it to the orchestrator (line 115–120). Add
`artifact_store=artifact_store` to the `LangGraphOrchestrator(...)` constructor call there.

### Anti-Patterns to Avoid

- **Do not raise in artifact persistence:** Follow the existing pattern — wrap in `try/except Exception`
  with `self._logger.warning(...)` to keep artifact persistence non-fatal. The main graph must never
  crash due to a Postgres artifact write failure.
- **Do not import ArtifactStore at module top in graph.py:** Use `TYPE_CHECKING` guard as with
  `MissionContextStore` — avoids circular import and runtime overhead.
- **Do not duplicate embedding generation:** The `ContextManager` already caches embeddings in
  `_embed_cache`. If the artifact value is different text than the goal, generate a fresh embedding
  per artifact (no caching needed for artifact values — they are written once).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Artifact persistence SQL | Custom INSERT | `ArtifactStore.upsert()` | Already handles ON CONFLICT, key_hash, embedding formatting |
| SHA-256 key hash | Custom hash | `_sha256_key()` inside `artifact_store.py` | Already implemented, tested |
| Embedding for artifact | Custom embed | `self._embedding_provider.embed_sync()` | Already injected into ContextManager |
| Zero-vector fallback | Custom check | Pattern `[0.0] * 384` already used in `_persist_mission_context` | Consistent with mission context embedding fallback |
| Pool-lifecycle management | Store-owned pool | `pool=None` graceful no-op + caller-managed pool | All project stores follow this pattern |

## Common Pitfalls

### Pitfall 1: Missing `artifacts` in `persist_ctx` dict

**What goes wrong:** `on_mission_complete()` builds `persist_ctx` at line 468, but the `ctx.artifacts`
list is not included. Calling `_persist_mission_context(persist_ctx)` without artifacts means the method
has no artifact data to persist.

**How to avoid:** Either (a) pass `artifacts=ctx.artifacts` in `persist_ctx` and iterate in
`_persist_mission_context`, or (b) iterate `ctx.artifacts` directly in `on_mission_complete()` and call
`self._artifact_store.upsert()` inline — after the existing `_persist_mission_context()` call.

### Pitfall 2: app.py creates ArtifactStore but doesn't forward it to orchestrator

**What goes wrong:** `app.py` at line 107 creates `artifact_store` and stores it on
`application.state`, but the `LangGraphOrchestrator(...)` call at line 115 does not pass
`artifact_store`. This means the FastAPI path never persists artifacts even when Postgres is configured.

**How to avoid:** Plan 07.5-02 (orchestrator param) and 07.5-03 (entrypoint wiring) must both
update `app.py` as part of the wiring, not just `run.py`/`user_run.py`.

### Pitfall 3: ScriptedProvider tests don't produce MissionContext artifacts

**What goes wrong:** The demo `ScriptedProvider`-based integration tests don't actually trigger
`on_tool_result()` → artifact extraction because tool calls need a live or scripted response cycle that
produces real `tool_result` records with populated `mission_contexts`.

**How to avoid:** The Postgres integration test (Plan 07.5-04) must directly call
`context_manager.on_tool_result()` and `context_manager.on_mission_complete()` rather than running
the full graph. Follow the same pattern as `test_mission_context_cascade.py` — construct a
`ContextManager` with a real `ArtifactStore(pool=pg_pool)` and call the lifecycle methods explicitly.

### Pitfall 4: `run_id` not passed into `on_mission_complete` → artifacts have `run_id="unknown"`

**What goes wrong:** `_persist_mission_context` reads `run_id` from the `persist_ctx` dict (line 298).
Artifacts must use the same `run_id`. Ensure `on_mission_complete` receives and passes `state["run_id"]`
to any `artifact_store.upsert()` call.

**How to avoid:** Use `state.get("run_id", "unknown")` — already present on `state` when
`on_mission_complete` is called (see graph.py line 477).

### Pitfall 5: Backward compat — `artifact_store=None` default must be preserved

**What goes wrong:** Any test that constructs `LangGraphOrchestrator()` or `ContextManager()` without
`artifact_store` keyword arg will break if the parameter has no default.

**How to avoid:** Both `__init__` signatures must use `artifact_store: ArtifactStore | None = None`
(keyword-only, default None). The 1230 existing tests pass `LangGraphOrchestrator` without this param.

### ArtifactStore.upsert() signature (verified from source)

```python
# Source: src/agentic_workflows/storage/artifact_store.py:62
def upsert(
    self,
    *,
    run_id: str,
    mission_id: str,
    key: str,
    value: str,
    source_tool: str,
    embedding: list[float],
) -> None:
```

All arguments are keyword-only. `mission_id` is `str` (not `int`) — use `str(mission_id)`.

### Iterating ArtifactRecord in _persist_mission_context (pattern)

```python
# Inside _persist_mission_context(), after the MissionContextStore.upsert():
if self._artifact_store is not None:
    artifacts = mission_context.get("artifacts", [])
    for art in artifacts:
        try:
            emb = self._embedding_provider.embed_sync(art["value"][:200]) \
                if self._embedding_provider is not None else [0.0] * 384
            self._artifact_store.upsert(
                run_id=run_id,
                mission_id=mission_id,
                key=art["key"],
                value=art["value"],
                source_tool=art["source_tool"],
                embedding=emb,
            )
        except Exception as exc:  # noqa: BLE001
            self._logger.warning("artifact upsert failed (non-fatal): %s", exc)
```

Note: `mission_context` is a dict (serialized `MissionContext.model_dump()`), so artifacts are dicts,
not `ArtifactRecord` objects. Access via `art["key"]`, `art["value"]`, `art["source_tool"]`.

### Postgres integration test pattern (verified from test_mission_context_cascade.py)

```python
# tests/integration/test_artifact_store_runtime.py

pytest.importorskip("psycopg_pool")

from agentic_workflows.context.embedding_provider import MockEmbeddingProvider
from agentic_workflows.orchestration.langgraph.context_manager import ContextManager, MissionContext
from agentic_workflows.storage.artifact_store import ArtifactStore

requires_postgres = pytest.mark.skipif(
    not os.environ.get("DATABASE_URL"),
    reason="DATABASE_URL not set -- skipping Postgres integration tests",
)

@requires_postgres
@pytest.mark.postgres
class TestArtifactStoreRuntime:
    def test_artifacts_persisted_after_mission_complete(self, pg_pool, clean_pg):
        provider = MockEmbeddingProvider()
        artifact_store = ArtifactStore(pool=pg_pool, embedding_provider=provider)
        cm = ContextManager(embedding_provider=provider, artifact_store=artifact_store)

        # Seed mission context with an artifact
        run_id = "test-run-art-01"
        state = {
            "run_id": run_id,
            "messages": [{"role": "system", "content": "sys"}],
            "mission_contexts": {
                "1": MissionContext(
                    mission_id=1,
                    goal="Write data to file",
                    status="pending",
                    step_range=(1, 3),
                ).model_dump()
            },
            "mission_reports": [{"mission_id": 1, "status": "completed", "mission": "Write data to file"}],
        }

        # Trigger on_tool_result to populate ctx.artifacts
        cm.on_tool_result(
            state, "write_file",
            result={"result": "Successfully wrote 10 characters to /tmp/out.txt"},
            args={"path": "/tmp/out.txt"},
            mission_id=1,
        )

        # Trigger on_mission_complete — should persist artifacts
        cm.on_mission_complete(state, mission_id=1)

        # Verify rows in mission_artifacts table
        with pg_pool.connection() as conn:
            rows = conn.execute(
                "SELECT key, value, source_tool FROM mission_artifacts WHERE run_id = %s",
                (run_id,)
            ).fetchall()

        assert len(rows) >= 1
        keys = [r[0] for r in rows]
        assert "file_path" in keys
```

## Open Questions

1. **Should artifact embeddings use artifact.value or artifact.key?**
   - What we know: `ArtifactStore.search()` uses embedding for semantic search of artifacts
   - What's unclear: Whether key (short, e.g. "file_path") or value (e.g. "/tmp/out.txt") is better
   - Recommendation: Embed `artifact.value[:200]` — the value contains the actual content; key is
     categorical. Consistent with the generic extractor's 200-char truncation.

2. **Does `app.py` also need updating for the orchestrator param?**
   - What we know: `app.py` creates `artifact_store` but does not pass it to orchestrator (line 115–120)
   - What's unclear: Which plan covers `app.py` — 07.5-02 (orchestrator param) or 07.5-03 (entrypoints)
   - Recommendation: Plan 07.5-03 covers CLI entrypoints; Plan 07.5-02 should also note `app.py` or
     the planner should assign `app.py` to 07.5-03 explicitly.
