The section must cover:
1. **What changed**: `_route_to_specialist()` now calls `self._executor_subgraph.invoke(exec_state)` for tool-type actions instead of `_execute_action()`. Both compiled subgraphs are cached as instance attributes in `__init__()`.
2. **Why**: Phase 3 built the subgraphs in isolation; Phase 4 connects them to the execution path. Caching prevents re-compilation overhead on every tool call in multi-mission runs.
3. **Which LangGraph classes implement it**: `CompiledStateGraph.invoke()` (returned by `StateGraph.compile()`); the wrapper-function pattern avoids `builder.add_node(compiled_subgraph)` which would require RunState key overlap.
4. **State merge detail**: `exec_tool_history` entries are copied into `RunState.tool_history` with `via_subgraph=True` tag and sequentially-assigned `call` indices to preserve `MissionAuditor._map_tool_history_to_missions()` cursor ordering.
5. **Evaluator position**: The evaluator subgraph is reserved for `_finalize()` time (not invoked mid-run) to avoid overwriting the canonical `audit_report` with partial-run data. The `eval_audit_report -> RunState.audit_report` merge path described in CONTEXT.md State Merge Strategy is deferred to Phase 5 per the Pitfall 4 analysis in RESEARCH.md — invoking the evaluator subgraph mid-run would produce partial-run audit data that `_finalize()` would then overwrite; the correct integration point is `_finalize()` itself, which already owns the canonical audit write.

Minimum 40 lines for the new section. Write in the same prose style as the existing WALKTHROUGH sections.
