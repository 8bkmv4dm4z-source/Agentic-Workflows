- Cache bounds: _cascade_cache / _embed_cache (200 entries max each)

State mutation spread across:
- context_manager.py — owns MissionContext lifecycle, eviction, injection
- planner_node.py — calls on_tool_result(), on_mission_complete(), auto-inject at step==1
- mission_tracker.py — refresh_mission_status(), subtask satisfaction, SUBTASK TICK

Four failure modes to document (from CONTEXT.md + RESEARCH.md):
1. Sub-task amnesia: after context eviction, agent loses awareness of completed sub-task results
   because summaries don't encode which sub-task produced which tool result
2. Result-to-subtask affiliation gap: tool_results in mission_reports stores {"tool":..., "result":...}
   but NOT which sub-task/step produced it; MissionContext.key_results is flat dict[str,str] not
   indexed by sub-task
3. Injection policy opacity: build_planner_context_injection() uses injected_mission_ids set for
   dedup but no caller can inspect what is currently injected without parsing state["messages"]
4. Scattered state mutation: mission_contexts dict in RunState is updated by on_mission_complete()
   in context_manager, but also read/written indirectly via mission_tracker.refresh_mission_status()
   and graph.py's _plan_next_action()

User framing: "the agent uses caching but lacks either the understanding or tools to process
multi-tasks written in human language and parse it — even after a few successful sub-phases
there is sub-task amnesia and lack of result chunk to sub-task affiliation."

    # ContextManager Architecture Audit

    ## Overview
    Brief summary of what ContextManager does and why an audit is needed.

    ## Current Architecture
    Describe the key components: MissionContext model fields, ContextManager class methods, state mutation touch points (context_manager.py, planner_node.py, mission_tracker.py). Reference actual file paths.

    ## Failure Mode 1: Sub-Task Amnesia
    - **Current behavior:** After context eviction (on_mission_complete evicts messages), the planner's next step has only the condensed summary. The summary (MissionContext.build_summary()) captures tool names and key results but not the sub-task that requested them. On resume, the planner cannot associate "sort_array result [1,2,3]" with "sub-task 1.2: sort the dataset".
    - **Root cause:** SubMissionContext exists in the model but on_tool_result() does not populate sub_mission_id on the recorded result. key_results dict uses tool name as key, not sub-task ID.
    - **Impact:** Agent re-issues previously completed sub-tasks after eviction. Mission completion check misses satisfied sub-tasks.
    - **Proposed fix:** On on_tool_result(), resolve the active sub-task from mission_tracker and store tool result under the sub-task key in a new sub_task_results: dict[str, list[str]] field on MissionContext. Include sub-task IDs in summary generation.

    ## Failure Mode 2: Result-to-Subtask Affiliation Gap
    - **Current behavior:** tool_results in mission_reports stores {"tool": ..., "result": ...} — no sub-task attribution. MissionContext.key_results is a flat dict keyed by tool name. When two sub-tasks call the same tool, only the last result is retained.
    - **Root cause:** key_results uses tool name as key (not sub-task ID + tool). The model schema doesn't enforce sub-task attribution.
    - **Impact:** Last-write-wins for repeated tool calls. Cannot reconstruct which sub-task produced which result.
    - **Proposed fix:** Change key_results to sub_task_results: dict[str, dict[str, str]] where outer key is sub-task ID, inner key is tool name. Update extract_artifacts() and build_summary() accordingly.

    ## Failure Mode 3: Injection Policy Opacity
    - **Current behavior:** build_planner_context_injection() tracks injected_mission_ids (instance set) for dedup. No external API exposes what is currently injected. The only way to verify injection is to parse state["messages"] for [Orchestrator] prefixed messages.
    - **Root cause:** The injection state is implicit (set membership) with no observable surface. No method returns "what context is currently visible to the planner".
    - **Impact:** Hard to debug context injection behavior in tests. Runaway re-injection was the root cause of the Phase 07.4 bug (fixed, but the opacity remains).
    - **Proposed fix:** Add get_injection_state() -> InjectionState dataclass method to ContextManager that returns: injected_mission_ids (set), last_injection_chars (int), cross_run_keys (set). Log state on each injection decision.

    ## Failure Mode 4: Scattered State Mutation
    - **Current behavior:** RunState["mission_contexts"] dict is written by on_mission_complete() in context_manager.py, read by build_specialist_context() in context_manager.py, and also read+mutated via mission_tracker.refresh_mission_status() which modifies mission status flags. planner_node.py calls both context_manager methods and mission_tracker in the same execution block.
    - **Root cause:** No single owner for mission context state. mission_tracker.py handles subtask satisfaction which should be ContextManager's concern.
    - **Impact:** Testing context behavior requires mocking multiple interacting components. A bug in mission_tracker can silently corrupt context injection.
    - **Proposed fix:** Consolidate all mission_contexts read/write through ContextManager as single owner. mission_tracker.refresh_mission_status() should call context_manager.on_subtask_complete() rather than mutating state directly.

    ## Proposed Architectural Changes (Phase 8.2 Scope)
    Enumerate the 4 concrete changes with implementation notes:
    1. SubTaskAwareToolResult schema — add sub_task_id to on_tool_result() signature; populate from mission_tracker active sub-task
    2. sub_task_results dict on MissionContext — replace flat key_results with sub-task-indexed structure
    3. get_injection_state() observable API — new method, no behavior change, adds testability
    4. Single-owner mutation contract — ContextManager.on_subtask_complete() consolidation

    ## Success Criteria for Phase 8.2
    List 4-6 observable behaviors that would confirm the redesign works:
    - After eviction, planner can identify which sub-task produced each cached result
    - Repeated tool calls by different sub-tasks produce separate result entries (no last-write-wins)
    - get_injection_state() returns accurate representation without parsing messages
    - mission_tracker tests can run without importing context_manager
    - All 823+ existing tests pass after changes

    ## Not In Scope (Phase 8.1)
    Explicitly state that NO code changes are made in this phase. The redesign executes in Phase 8.2.
