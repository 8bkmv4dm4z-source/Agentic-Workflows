### Pattern 5: ContextManager Architecture Audit

**What:** Produce a `08.1-context-audit.md` document — architecture analysis only, no code changes.

The ContextManager (1054 lines in `context_manager.py`) implements:
- `MissionContext` Pydantic model with `artifacts`, `sub_missions`, `key_results`, `summary`, `step_range`
- `ContextManager` class with `on_tool_result()`, `on_mission_complete()`, `build_planner_context_injection()`, `build_specialist_context()`, `_persist_mission_context()`, `persist_partial_missions()`
- Cross-run cascade injection via `MissionContextStore.query_cascade()`
- Large result truncation via `_LARGE_RESULT_THRESHOLD`
- Cache bounds via `_cascade_cache` / `_embed_cache` (200 entries max)

State mutation is spread across:
- `context_manager.py` — owns MissionContext lifecycle, eviction, injection
- `graph.py` / `planner_node.py` — calls `on_tool_result()`, `on_mission_complete()`, auto-inject at step 1
- `mission_tracker.py` — owns `refresh_mission_status()`, subtask satisfaction, SUBTASK TICK

The audit document must cover:
1. **Sub-task amnesia** — after context eviction, agent loses awareness of completed sub-task results because summaries don't encode which sub-task produced which tool result
2. **Result-to-subtask affiliation gap** — `tool_results` in `mission_reports` stores `{"tool": ..., "result": ...}` but not which sub-task/step produced it; `MissionContext.key_results` is a flat `dict[str, str]` not indexed by sub-task
3. **Injection policy opacity** — `build_planner_context_injection()` uses `injected_mission_ids` set for dedup but no caller can inspect what is currently injected without parsing `state["messages"]`
4. **Scattered state mutation** — `mission_contexts` dict in RunState is updated by `on_mission_complete()` in context_manager, but also read/written indirectly via `mission_tracker.refresh_mission_status()` and graph.py's `_plan_next_action()`

- **Trusting the audit without verification.** At least 3 of the 5 audit "gaps" are already resolved. Verify each in live source.

## Common Pitfalls

### Pitfall 1: Trusting the Audit Document as Current Ground Truth
**What goes wrong:** Creating tasks to fix gaps that are already resolved, wasting time and potentially introducing regressions.
**Why it happens:** The v1.5-MILESTONE-AUDIT.md was written 2026-03-11; multiple fixes were applied after that date.
**How to avoid:** Grep each claimed gap before creating a task. If the gap is already closed, skip to verifying the milestone audit re-run passes.
**Warning signs:** Any item from the audit that references specific line numbers (e.g., "line 2778") is suspicious — the code has been decomposed since then.

### Pitfall 2: Removing route_by_intent() Without Test Cleanup
**What goes wrong:** Deleting the method causes 4 tests in `test_model_router_wiring.py` to fail.
**Why it happens:** The audit says "no active callers in src/" — true, but tests ARE callers.
**How to avoid:** Run `grep -rn "route_by_intent" src/ tests/` before any deletion. Remove or update the tests in the same commit.
**Warning signs:** Test file at `tests/unit/test_model_router_wiring.py:181–238` has a dedicated "Deprecated route_by_intent backward-compat tests" section.

### Pitfall 3: Treating route_by_signals Double Call as Redundant
**What goes wrong:** Removing one of the `route_by_signals` calls breaks either routing-decision tracking or the hard-timeout provider selection.
**Why it happens:** The audit referenced lines in the old monolithic graph.py; after decomposition the two calls live in separate files with separate purposes.
**How to avoid:** Read `planner_node.py:260–270` (tracks structural_health routing_decisions) vs `planner_helpers.py:628–635` (_generate_with_hard_timeout provider selection). They are not equivalent.
**Warning signs:** Removing either call would require rethinking how routing_decisions are counted in structural_health.

### Pitfall 4: Nyquist Validation Generates Tests That Fail
**What goes wrong:** `/gsd:validate-phase 07.1` generates Wave 0 stubs — these start as `NotImplementedError` stubs (RED) and need to be made green.
**Why it happens:** The validator generates stubs but implementation may be missing or test targets may not yet exist.
**How to avoid:** After running `/gsd:validate-phase`, run the full test suite. Any failing stubs need the corresponding implementation or need to be adjusted to match existing behavior.
**Warning signs:** Phase 07.1 VALIDATION.md has 11 Wave 0 test stubs all marked `No — W0`.

### Pitfall 5: Milestone Audit Re-Run Criteria
**What goes wrong:** Declaring the phase complete before verifying the audit itself would return `status: passed`.
**Why it happens:** The audit scores (requirements, phases, integration, flows) each have specific criteria.
**How to avoid:** The audit's `status: gaps_found` was driven by 3 items: LGUP-02 stale verification, app.py fallback gap, and Nyquist partial compliance. After Phase 08.1, at minimum: (1) LGUP-02 verification must be `passed`, (2) app.py must have fallback_provider wired, (3) Nyquist must improve (exact threshold is best-effort, not blocking).
