# Phase 07.6 Post-Phase Notes

## Quick Task 5: Tool Schema Enforcement (2026-03-10)

**Motivation:** Qwen3-8B and similarly weak local models hallucinate tool arg names when the compact
prompt lists bare tool names with no schema (`classify_intent` instead of `classify_intent(text)`).
Observed symptom: `classify_intent` called with `payload.text` instead of `text`, looping indefinitely.

**Two complementary fixes implemented:**

### Part A — Arg Signatures in Compact Prompt (soft enforcement)

`Tool.required_args()` added to `src/agentic_workflows/tools/base.py`:
- Parses the description string's "Required args: X (type), Y (type)" section via regex
- Returns `["text"]`, `["pattern"]`, `["path"]`, etc.

Compact prompt tool listing changed from:
```
classify_intent, search_files, read_file, write_file, ...
```
to:
```
classify_intent(text), search_files(pattern), read_file(path), write_file(path, content), ...
```

Implementation: `_tool_sig()` inner function in `graph.py`, called when building `tool_names_line`
in the compact branch.

### Part B — JSON Schema response_format (hard enforcement)

`ChatProvider.generate()` protocol extended with `response_schema: dict | None = None`.

`LangGraphOrchestrator._build_action_json_schema()` builds an `anyOf` JSON schema from the live
tool registry — one variant per tool with `{"const": name}` on `tool_name` and required arg
properties. Includes `finish` and `clarify` variants.

Provider behavior:
- **LlamaCpp** — uses `response_schema` as `response_format` when grammar is disabled (`LLAMA_CPP_GRAMMAR=false`), replaces fragile GBNF
- **OpenAI** — uses `response_schema` if provided, else falls back to `_OPENAI_ACTION_RESPONSE_FORMAT`
- **Ollama / Groq** — accept param, ignore it (no json_schema support)
- **ScriptedProvider** — accepts param, ignores it (tests unaffected)

Schema cached as `self._action_json_schema` in `__init__` after `build_tool_registry()`. Both
`_generate_with_hard_timeout()` call sites pass `response_schema=self._action_json_schema`.
