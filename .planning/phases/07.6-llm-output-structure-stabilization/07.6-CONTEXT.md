# Phase 07.6: LLM Output Structure Stabilization - Context

**Gathered:** 2026-03-09
**Status:** Ready for planning
**Source:** PRD Express Path (user-provided stabilization plan)

<domain>
## Phase Boundary

This phase mechanically enforces LLM output structure before multi-mission agent teams deploy weaker executor models. It fixes the phi4 context overflow blocker (Obstacle 0), adds provider-aware compact prompts, re-enables GBNF grammar for llama-cpp, instruments the fallback parser, converts handoff TypedDicts to Pydantic, persists chunked-read cursors to survive context eviction, and adds structural health metrics to the audit report.

**NOT in scope:** PlanAction schema (Phase B in the plan — deferred until phi4 is stable).

</domain>

<decisions>
## Implementation Decisions

### Obstacle 0 Fix — Provider-Aware Prompt Sizing (CRITICAL, phi4 blocker)

- Add `context_size() -> int` method to `ChatProvider` protocol and all four provider classes
  - `LlamaCppChatProvider`: read `LLAMA_CPP_N_CTX` env var or query `/props` endpoint (default 8192)
  - `GroqChatProvider`: return 32768
  - `OpenAIChatProvider`: return 128000
  - `OllamaChatProvider`: return `OLLAMA_NUM_CTX` env var if set, else 32768
- Add `_select_prompt_tier(context_size: int) -> Literal["compact", "full"]` in graph.py
  - `compact` when `context_size <= 10000`
  - `full` otherwise
- `LangGraphOrchestrator.__init__` calls `provider.context_size()` and stores the tier
- `_build_system_prompt()` uses tier to select compact vs full tool list and directives

### Compact System Prompt — C1

- `supervisor.md`: add `## COMPACT` section (≤ 15 lines) at the top
  - Role + schema (3 lines)
  - 3 rules (python3, run_bash, no markdown)
  - 2 short examples
- `graph.py` `_build_system_prompt()`: when tier=compact, inject only the COMPACT section, not full directive text

### Compact Tool List — C2

- When `tier == "compact"`: inject tool names only — one comma-separated line, no arg signatures
- When `tier == "full"`: inject full arg signatures (current behavior)

### Environment Facts Injection — C3

- Both tiers inject an `<env>` block with:
  - `python3 is available (not python)`
  - `Working dir: {cwd}`
- Fixes `python: not found` loop seen in logs

### Re-Enable GBNF Grammar — A1

- `.env.example` line 37: clarify comment — grammar is enabled by default, the line shows how to DISABLE it
- `provider.py` `LlamaCppChatProvider.__init__`: `_grammar_enabled` already defaults to true when env var absent — no code change needed, just env.example comment update

### Fallback Parser Instrumentation — A2

- `action_parser.py` `parse_action_json()`: when `extract_first_json_object()` is used (fallback path), emit:
  - `WARNING: PARSER FALLBACK used=extract_first_json step=<step> model_output=<first 200 chars>`
- Also log prose prefix text when it exists before `{`
- Add `step` parameter to `parse_action_json()` (optional, default 0) so callers can pass step number

### Handoff Pydantic Conversion — A3

- `handoff.py`: convert `TaskHandoff` and `HandoffResult` from TypedDict to Pydantic BaseModel
  - `model_config = ConfigDict(extra="forbid")` on both
  - Keep `create_handoff()` and `create_handoff_result()` factory functions (update to use Pydantic)
  - Must remain serializable as dicts for RunState (use `model_dump()`)
- Update all callsites that construct these directly

### Sub-Task Cursor Persistence — E1–E4

- `mission_context_store.py`:
  - Add `upsert_cursor(run_id, plan_step_id, mission_id, tool_name, key, cursor, total)` method
  - Add `get_cursor(run_id, plan_step_id, mission_id, tool_name, key) -> int | None` method
  - Add `get_active_cursors(run_id) -> list[dict]` method (returns all in-progress cursors)
  - SQLite: in-memory dict fallback when pool=None; Postgres: `sub_task_cursors` table
- `graph.py` executor result handler:
  - After `read_file_chunk` returns `has_more=True`: call `upsert_cursor()` with `next_offset`
  - After `read_file_chunk` returns `has_more=False`: delete/clear the cursor
- `context_manager.py` after sliding window eviction:
  - Query `get_active_cursors(run_id)` and inject terse hint messages:
    `[Orchestrator] Chunked read in progress: read_file_chunk path=<X> next_offset=<N> of <total> lines. Continue from offset=<N>.`
- `graph.py` seen_tool_signatures check:
  - When a `read_file_chunk` call matches an active cursor hint, allow it (bypass duplicate detection)
  - Tag cursor-resumption actions with `__cursor_resume: True` to bypass `seen_tool_signatures`

### Structural Health Metrics — D2

- `state_schema.py`: add `structural_health` dict to RunState (or populate in `_finalize()`)
- `graph.py`: track counters:
  - `json_parse_fallback` — incremented each time fallback parser is used
  - `schema_mismatch` — incremented each time Pydantic rejects a parsed action
- Expose in `audit_report["structural_health"]` after every run

### Implementation Order (from plan's Revised Execution Order)

1. C0/C1/C2/C3 — provider context_size + compact prompt (unblocks phi4)
2. E1–E4 — sub-task cursor persistence (fixes chunk-restart duplicate-kill loop)
3. A1 — env.example grammar comment update
4. A2 — fallback parser instrumentation
5. A3 — handoff Pydantic conversion
6. D2 — structural health metrics

### Claude's Discretion

- SQLite cursor storage: use a simple in-memory dict on the MissionContextStore instance (no disk table) since SQLite is dev-only — Postgres gets a real table
- Cursor key format: `f"{run_id}:{mission_id}:{tool_name}:{path}"` for simple dedup
- `parse_action_json()` step parameter: thread it through all callers in graph.py or add it as an optional kwarg with default=0
- Backward compat: all changes are additive — existing behavior unchanged when tier=full (default for Groq/OpenAI/Ollama with large context)

### Compact Supervisor COMPACT Section Example
```
## COMPACT
You emit exactly one JSON action per response. No markdown, no prose, no XML. Pure JSON only.
Schema: {"action":"tool","tool_name":"X","args":{...}} | {"action":"finish","answer":"X"} | {"action":"clarify","question":"X"}
Rules: Use python3 not python. Use run_bash for shell commands. One tool per response.
Example: {"action":"tool","tool_name":"run_bash","args":{"command":"python3 -c 'print(1)'","timeout":10}}
Example: {"action":"finish","answer":"Task complete: wrote fib50.txt with 50 numbers"}
```

### LlamaCpp context_size() implementation
```python
def context_size(self) -> int:
    n_ctx = int(os.getenv("LLAMA_CPP_N_CTX", "8192"))
    return n_ctx
```
