## Summary

Phase 7.2 addresses 10 discrete issues discovered during an architectural review. The issues span three severity tiers: two correctness bugs (dual tool execution in `_route_to_specialist`, `_active_callbacks` data race), five critical bottlenecks (SQLite WAL/persistent connection, `seen_tool_signatures` set conversion, pipeline_trace/handoff list caps), and three structural/safety improvements (`_ANNOTATED_LIST_FIELDS` auto-derivation, `prepare_state()` extraction, `run_bash` guard, memoize prompt removal, tool contract tests).

All 10 changes are internal -- no observable behavior change from the user's perspective. The existing 657-test suite provides extensive regression coverage. Each fix is well-scoped, touching specific files with clear before/after boundaries. The primary risk is subtle interactions between the `seen_tool_signatures` set conversion and the `_ANNOTATED_LIST_FIELDS` auto-derivation, since both touch the RunState type contract.

**Primary recommendation:** Execute in strict wave order (correctness first, bottlenecks second, structural last). Run the full 657-test suite after each wave merge to catch regressions early.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Wave 1 -- Correctness Bugs (highest priority, fix first)**
- **Dual-execution of side-effectful tools**: `_route_to_specialist` (graph.py lines ~1458-1462) fires the tool AND `_execute_action` fires it again -- two writes, two HTTP calls, two side effects. Must eliminate the duplicate execution path.
- **`self._active_callbacks` data race**: The singleton `LangGraphOrchestrator` overwrites `self._active_callbacks = []` on every `run()` call. Concurrent requests clobber each other's Langfuse spans. Must scope `_active_callbacks` to the run context, not the instance.

**Wave 2 -- Critical Bottlenecks**
- **SQLite WAL mode + persistent connection**: `SQLiteCheckpointStore` creates a new connection per `save()` call and does not enable WAL mode. Fix: enable WAL on first connect, reuse a single connection across saves.
- **`seen_tool_signatures` -> set**: Currently `Annotated[list, operator.add]` -- triggers O(n) linear scan on every dedup check and pays reducer overhead on every graph edge. Convert to a plain `set` field (non-annotated, non-reduced). Update all read/write sites.
- **Cap `pipeline_trace`, `handoff_queue`, `handoff_results`**: These lists grow unboundedly. Apply oldest-first eviction with a configurable cap (default: 500 for pipeline_trace, 50 for queues).

**Wave 3 -- Structural / Safety**
- **Derive `_ANNOTATED_LIST_FIELDS` automatically**: The frozenset in graph.py is maintained manually and drifts when fields are added/removed. Derive it at class-load time from `RunState.__annotations__` by inspecting for `Annotated[list, ...]` shape.
- **Consolidate state initialization**: `run()` and the SSE route handler both initialize RunState inline. Extract a `prepare_state()` function called by both -- single source of truth.
- **Security guardrails default-on**: `run_bash` currently requires no opt-in. Add `P1_BASH_ENABLED=true` env-var check so bash execution is explicitly opted into. Default: disabled.
- **Remove `memoize` from planner tool list**: Planner currently lists `memoize` as a callable tool. Auto-handle memoization in the post-execution step instead. Remove from tool options visible to the LLM.
- **Parametrized tool contract test**: Add a pytest parametrize fixture covering all registered tools -- verifying each has a name, description, and callable `execute` method, and that `execute` returns a dict (or raises a typed error).

#### W1-1: Dual-Execution Fix
**Current bug (graph.py ~1458-1462):**
```python
# Line 1458: Subgraph invoked (tool executes inside subgraph)
self._executor_subgraph.invoke(exec_state, config={"callbacks": self._active_callbacks})
# Line 1462: _execute_action called again (tool executes AGAIN in main graph)
state = self._execute_action(state)
```
**Fix pattern:** Remove the `self._executor_subgraph.invoke()` call. The subgraph invocation was added for Langfuse observability (recording subgraph node transitions) but the actual tool execution must go through `_execute_action()` only, because `_execute_action()` handles arg normalization, duplicate detection, auto-memo-lookup, content validation, and mission attribution. The subgraph `invoke()` produces a throwaway result -- it cannot safely execute side-effectful tools.

**Alternative considered:** Remove `_execute_action()` and use only the subgraph. Rejected because the subgraph does not implement the full execution pipeline (dedup, memo policy, content validation, mission attribution). This would require duplicating ~200 lines of pipeline logic into the subgraph.

**Recommended approach:** Remove the `self._executor_subgraph.invoke()` line. Also remove the `exec_state` dict construction block (lines 1438-1456) since the local variable is no longer used. Keep the logging/trace comment explaining why. If Langfuse subgraph node transition tracing is needed later, wire it via `_emit_trace()` instead, which is side-effect-free.

**Test impact:** `test_subgraph_routing.py::test_tool_action_sets_via_subgraph_tag` passes because `_execute_action()` still runs and tags `via_subgraph=True` (lines 1465-1466). The test checks for the tag, not for subgraph invocation.

**Confidence:** HIGH -- direct code reading confirms the dual execution.

#### W1-2: _active_callbacks ContextVar
**Current bug (graph.py ~162, ~403):**
```python
# __init__ (line 162):
self._active_callbacks: list[Any] = []

# run() (line 403):
self._active_callbacks: list[Any] = []
_handler = get_langfuse_callback_handler()
if _handler is not None:
    self._active_callbacks = [_handler]
```
The singleton orchestrator instance is shared across concurrent requests (FastAPI). `self._active_callbacks` is an instance attribute overwritten at `run()` start. If two requests call `run()` concurrently, request B's callbacks clobber request A's.

**Fix pattern:** Use `contextvars.ContextVar` at module level:
```python
import contextvars
_active_callbacks_var: contextvars.ContextVar[list[Any]] = contextvars.ContextVar(
    "_active_callbacks", default=[]
)
```
In `run()`, set the ContextVar:
```python
callbacks = [_handler] if _handler else []
_active_callbacks_var.set(callbacks)
```
At all read sites, read from ContextVar:
```python
config={"callbacks": _active_callbacks_var.get([])}
```

**Why ContextVar over alternatives:**
- Thread-local (`threading.local`): Does not work with asyncio -- FastAPI uses asyncio, and thread-local state is not propagated across `await` boundaries. `contextvars.ContextVar` IS propagated.
- Run-local dict keyed by run_id: Requires threading run_id into every site. ContextVar is cleaner.
- ContextVar is the Python 3.7+ standard for per-task state in asyncio.

**Read sites to update (4 total):**
1. `graph.py:162` -- `__init__` initialization (remove)
2. `graph.py:403-406` -- `run()` callback setup (use ContextVar.set)
3. `graph.py:449` -- `self._compiled.invoke(config={"callbacks": ...})`
4. `graph.py:1458` -- `self._executor_subgraph.invoke(config={"callbacks": ...})` (removed by W1-1)

After W1-1 removes line 1458, only 3 sites remain (162, 403-406, 449).

Also update the SSE route handler in `api/routes/run.py:122-125` where `orchestrator._active_callbacks` is set directly.

**Confidence:** HIGH -- `contextvars.ContextVar` is well-documented Python stdlib behavior.

#### W2-3: SQLiteCheckpointStore WAL + Persistent Connection
**Current code (checkpoint_store.py):**
```python
def _connect(self) -> sqlite3.Connection:
    conn = sqlite3.connect(self.db_path)
    conn.row_factory = sqlite3.Row
    return conn

def save(self, ...) -> None:
    with self._connect() as conn:  # new connection per call
        conn.execute(...)
```
Every `save()`, `load_latest()`, `list_checkpoints()`, and `list_runs()` creates a new connection.

**Reference implementation:** `storage/sqlite.py` (SQLiteRunStore) already does this correctly:
```python
def __init__(self, ...):
    self._lock = threading.Lock()
    self._conn = sqlite3.connect(db_path, check_same_thread=False)
    self._conn.execute("PRAGMA journal_mode=WAL")
    self._conn.execute("PRAGMA busy_timeout=5000")
```

**Fix pattern:** Mirror `SQLiteRunStore`:
1. In `__init__`: create persistent `self._conn` with `check_same_thread=False`
2. Enable `PRAGMA journal_mode=WAL` and `PRAGMA busy_timeout=5000`
3. Add `self._lock = threading.Lock()`
4. Replace `self._connect()` context managers with `with self._lock:` + `self._conn.execute()`
5. Add `close()` method
6. Remove `_connect()` method

**Confidence:** HIGH -- `SQLiteRunStore` in the same codebase proves the pattern works.

#### W2-4: seen_tool_signatures Set Conversion
**Current code (state_schema.py:85):**
```python
seen_tool_signatures: Annotated[list[str], operator.add]
```
This is a list used for O(n) `in` checks at graph.py:1683, and appended to at graph.py:1735. It also participates in the `_ANNOTATED_LIST_FIELDS` frozenset (graph.py:86) and `_sequential_node` zeroing.

**Fix pattern:**
1. In `state_schema.py`: Change to `seen_tool_signatures: set[str]` (no Annotated, no reducer)
2. In `new_run_state()`: Change default from `[]` to `set()`
3. In `ensure_state_defaults()`: Change default from `[]` to `set()`, add list->set conversion for deserialized state
4. In `graph.py:1683`: `if signature in state["seen_tool_signatures"]` -- already O(1) with set
5. In `graph.py:1735`: `state["seen_tool_signatures"].add(signature)` (was `.append()`)
6. Remove `"seen_tool_signatures"` from `_ANNOTATED_LIST_FIELDS` (or rely on W3-6 auto-derivation)
7. In the Anthropic ToolNode dedup wrapper (graph.py:310): update default from `[]` to `set()`

**Serialization concern:** JSON serialization for checkpointing. `json.dumps(state)` will fail on `set`. Need a custom default handler:
```python
def _json_default(obj):
    if isinstance(obj, set):
        return sorted(obj)  # deterministic ordering for reproducibility
    return str(obj)
```
Update `checkpoint_store.py` save method to use this handler. In `ensure_state_defaults()`, add list->set conversion:
```python
if "seen_tool_signatures" in state_dict:
    val = state_dict["seen_tool_signatures"]
    if isinstance(val, list):
        state_dict["seen_tool_signatures"] = set(val)
```

**LangGraph reducer interaction:** Removing `Annotated[..., operator.add]` means LangGraph will use plain assignment (last-write-wins) for this field. This is correct for sequential execution (current usage). For future parallel `Send()` branches, sets would need a custom reducer (e.g., `operator.or_`). This is acceptable since parallel execution is v2 scope and deferred.

**`_sequential_node` impact:** After removing `seen_tool_signatures` from the annotated list fields, `_sequential_node` no longer zeroes it. This is correct because the field is no longer reduced.

**Confidence:** HIGH -- straightforward type change with well-understood serialization boundary.

#### W2-5: Cap pipeline_trace, handoff_queue, handoff_results
**Current code locations:**
- `pipeline_trace`: stored in `state["policy_flags"]["pipeline_trace"]` (list of dicts). Written by `_emit_trace()` at graph.py:475 and `context_manager.py:490`. Read by `run_ui.py:198`.
- `handoff_queue`: `state["handoff_queue"]` (list of dicts). Written at graph.py:1410. Read for routing decisions.
- `handoff_results`: `state["handoff_results"]` (list of dicts). Written at graph.py:1483.

**Recommended cap values:**
- `pipeline_trace`: 500 (diagnostic, rarely read during execution; typical run: 20-100 steps x 2 events = 40-200)
- `handoff_queue`: 50 (consumed during routing, older entries are stale)
- `handoff_results`: 50 (audit trail, most recent matter most)

**Implementation pattern:** Constants at module level in graph.py:
```python

_HANDOFF_QUEUE_CAP = 50
_HANDOFF_RESULTS_CAP = 50
```
Eviction: oldest-first (delete front of list when over cap):
```python
trace.append({...})
if len(trace) > _PIPELINE_TRACE_CAP:
    del trace[:len(trace) - _PIPELINE_TRACE_CAP]
```

Also add cap enforcement in `context_manager.py:490` where pipeline_trace is appended.

**Rationale for hardcoded constants (not env vars):** These are internal diagnostic limits, not user-facing configuration. Env vars add complexity for zero user benefit. If tuning is needed later, constants are easy to promote.

**Confidence:** HIGH -- simple list truncation with clear semantics.

#### W3-6: Auto-derive _ANNOTATED_LIST_FIELDS
**Current code (graph.py:85-87):**
```python
_ANNOTATED_LIST_FIELDS: frozenset[str] = frozenset(
    {"tool_history", "memo_events", "seen_tool_signatures", "mission_reports"}
)
```
After W2-4 removes `seen_tool_signatures` from annotated fields, this must be updated. Instead of manual maintenance, derive at module load time.

**Verified derivation pattern (tested against actual codebase):**
```python
import typing
import operator
from agentic_workflows.orchestration.langgraph.state_schema import RunState

def _derive_annotated_list_fields() -> frozenset[str]:
    hints = typing.get_type_hints(RunState, include_extras=True)
    fields = set()
    for name, hint in hints.items():
        if typing.get_origin(hint) is typing.Annotated:
            args = typing.get_args(hint)
            base = args[0]
            metadata = args[1:]
            if typing.get_origin(base) is list and operator.add in metadata:
                fields.add(name)
    return frozenset(fields)

_ANNOTATED_LIST_FIELDS = _derive_annotated_list_fields()
```

**Runtime verification result:** Running the above against the current codebase produces `{'tool_history', 'memo_events', 'seen_tool_signatures', 'mission_reports'}`. After W2-4 removes the annotation from `seen_tool_signatures`, it will correctly produce `{'tool_history', 'memo_events', 'mission_reports'}`.

**Note:** The `from __future__ import annotations` at the top of state_schema.py makes annotations lazy strings. `typing.get_type_hints(..., include_extras=True)` resolves them correctly. The derivation function is in graph.py which imports state_schema.py -- no circular import risk.

**Confidence:** HIGH -- verified with runtime test on actual codebase.

### Anti-Patterns to Avoid

- **DO NOT remove `_execute_action()` for the dual-execution fix.** The subgraph invoke is the redundant path, not the main execution pipeline.
- **DO NOT use `threading.local()` for `_active_callbacks`.** It does not propagate across `await` boundaries in asyncio. Use `contextvars.ContextVar`.
- **DO NOT add `set` directly to RunState TypedDict without handling JSON serialization.** LangGraph checkpointer and the SQLite checkpoint store use `json.dumps()`.
- **DO NOT make cap constants configurable via env vars.** Internal limits should be constants, not user-facing config.
- **DO NOT remove `memoize` from `self.tools` registry.** Only remove from the LLM-visible prompt. Internal code calls `self.tools["memoize"].execute(...)`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-request context isolation | Thread-local dict | `contextvars.ContextVar` | Built into Python 3.7+; works with asyncio |
| SQLite connection pooling | Custom pool | Single persistent `sqlite3.connect(check_same_thread=False)` + `threading.Lock` | Proven pattern in same codebase (`SQLiteRunStore`) |
| Annotation introspection | Manual string parsing | `typing.get_type_hints(cls, include_extras=True)` + `typing.get_origin/get_args` | Standard library; handles `from __future__ import annotations` |
| JSON set serialization | Custom encoder class | `default=` handler with `sorted(x) if isinstance(x, set)` | One-liner, deterministic output |
| Test parametrization over registry | Loop-based test | `pytest.mark.parametrize` | Better error reporting, individual test identity |

## Common Pitfalls

### Pitfall 1: seen_tool_signatures JSON Serialization
**What goes wrong:** After converting to `set`, `json.dumps(state)` raises `TypeError: Object of type set is not JSON serializable` in `checkpoint_store.save()`.
**Why it happens:** The existing `default=str` converts sets to their string repr (`"{'sig1', 'sig2'}"`) which cannot be round-tripped.
**How to avoid:** Update `json.dumps` default handler in checkpoint_store.py to convert sets to sorted lists. In `ensure_state_defaults()`, convert list back to set after deserialization.
**Warning signs:** Tests that save and load checkpoints will catch this immediately.

### Pitfall 2: _ANNOTATED_LIST_FIELDS Stale Between Waves
**What goes wrong:** If W2-4 (set conversion) lands before W3-6 (auto-derivation), there's a window where `_ANNOTATED_LIST_FIELDS` still contains `"seen_tool_signatures"` but the field is no longer annotated.
**Why it happens:** The manual frozenset is not updated between waves.
**How to avoid:** In W2-4, manually remove `"seen_tool_signatures"` from the frozenset. W3-6 then replaces the manual frozenset with auto-derivation.
**Warning signs:** `_sequential_node` will try to zero a set field with `result[field] = []`, overwriting the set with an empty list.

### Pitfall 3: _active_callbacks SSE Route
**What goes wrong:** The SSE route handler in `api/routes/run.py:122-125` directly sets `orchestrator._active_callbacks`. After the ContextVar fix, this code will write to a now-unused instance attribute.
**Why it happens:** The SSE route was written before the ContextVar migration.
**How to avoid:** Update the SSE route to use the same ContextVar. Search for ALL `_active_callbacks` references across the codebase.

### Pitfall 4: run_bash Test Breakage
**What goes wrong:** All 5 existing `test_run_bash.py` tests fail after adding the `P1_BASH_ENABLED` guard.
**Why it happens:** Tests don't set the env var, so `execute()` returns the disabled error immediately.
**How to avoid:** Update the `tool` fixture to set `P1_BASH_ENABLED=true` via `monkeypatch`. Add a dedicated test for the disabled case without the env var.

### Pitfall 5: prepare_state() Callback Setup
**What goes wrong:** Extracting `_prepare_state()` from `run()` may accidentally include or exclude the callback setup or graph invocation.
**Why it happens:** The boundary between "state preparation" and "run execution" is fuzzy.
**How to avoid:** `_prepare_state()` should only build and return the RunState (including initial checkpoint save). Callback setup (ContextVar.set) and graph invocation remain in `run()` and the SSE handler.

### Pitfall 6: Memoize Still Needed Internally
**What goes wrong:** Removing `memoize` from `self.tools` breaks `_execute_action()` which calls `self.tools["memoize"].execute(...)` at line 1920.
**Why it happens:** Confusing "remove from LLM prompt" with "remove from registry."
**How to avoid:** Only remove `memoize` from the tool list string in the system prompt. Keep it in `self.tools` registry.

### Pitfall 7: _dedup_then_tool_node Default
**What goes wrong:** After converting `seen_tool_signatures` to set, `graph.py:310` has `state.get("seen_tool_signatures", [])`. The `in` operator works on both list and set, but the default `[]` means a new empty list is allocated on each call when the key is missing.
**Why it happens:** Default wasn't updated to match new type.
**How to avoid:** Change default from `[]` to `set()`: `state.get("seen_tool_signatures", set())`.

### Auto-derive _ANNOTATED_LIST_FIELDS

```python
# Source: Verified against actual RunState annotations in this codebase
import typing
import operator

def _derive_annotated_list_fields(state_cls: type) -> frozenset[str]:
    """Derive fields with Annotated[list[T], operator.add] at class-load time."""
    hints = typing.get_type_hints(state_cls, include_extras=True)
    fields: set[str] = set()
    for name, hint in hints.items():
        if typing.get_origin(hint) is typing.Annotated:
            args = typing.get_args(hint)
            base, *metadata = args
            if typing.get_origin(base) is list and operator.add in metadata:
                fields.add(name)
    return frozenset(fields)
```

### SQLite WAL + Persistent Connection (from SQLiteRunStore)
```python
# Source: src/agentic_workflows/storage/sqlite.py (existing reference implementation)
import sqlite3
import threading

class SQLiteCheckpointStore:
    def __init__(self, db_path: str = ".tmp/langgraph_checkpoints.db") -> None:
        self.db_path = Path(db_path)
        self.db_path.parent.mkdir(parents=True, exist_ok=True)
        self._lock = threading.Lock()
        self._conn = sqlite3.connect(str(db_path), check_same_thread=False)
        self._conn.row_factory = sqlite3.Row
        self._conn.execute("PRAGMA journal_mode=WAL")
        self._conn.execute("PRAGMA busy_timeout=5000")
        self._initialize_schema()

    def save(self, ...) -> None:
        with self._lock:
            self._conn.execute(...)
            self._conn.commit()

    def close(self) -> None:
        self._conn.close()
```
