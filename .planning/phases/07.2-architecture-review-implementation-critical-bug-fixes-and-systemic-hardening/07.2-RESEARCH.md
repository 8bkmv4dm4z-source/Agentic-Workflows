## Architecture Patterns

### Wave 1 Fix 1: Dual-Execution in `_route_to_specialist`

**What goes wrong:** `_route_to_specialist` at lines 1457-1462 in `graph.py` calls `self._executor_subgraph.invoke(exec_state, ...)` followed by `self._execute_action(state)`. The subgraph invoke runs executor logic (including tool execution inside the subgraph), and `_execute_action` then fires the same tool again through the main pipeline.

**Confirmed execution path:**
```
graph.py:1458  self._executor_subgraph.invoke(exec_state, ...)   ← subgraph fires tool
graph.py:1462  state = self._execute_action(state)               ← main pipeline fires same tool again
```

**Fix pattern:** Remove the `self._executor_subgraph.invoke()` call, or convert the subgraph call to a tracing-only record without real tool execution. Since `_execute_action` already applies full pipeline (arg normalization, dedup, auto-memo, content validation, mission attribution) and `exec_state` is built but never read back into `state`, the subgraph invocation is the redundant call. The fix is to **remove the `self._executor_subgraph.invoke()` call** and preserve only the `_execute_action` pipeline path, which already tags tool_history entries with `via_subgraph=True`.

**Implication for tests:** Any test asserting "subgraph was called" must be updated; tests asserting "tool executed once" now pass.

### Wave 1 Fix 2: `_active_callbacks` Data Race

**What goes wrong:** `self._active_callbacks` is an instance variable on the singleton `LangGraphOrchestrator`. Two concurrent `run()` calls both do `self._active_callbacks = []` and then `self._active_callbacks = [_handler]`. Each call may read the other's handler list mid-run.

**Confirmed sites:**
- `graph.py:403-406` — `run()` method resets instance field
- `routes/run.py:122-125` — SSE route handler also resets the same field

**Fix options (discretion area):**

| Option | Mechanism | Thread-safe? | Async-safe? | Complexity |
|--------|-----------|-------------|------------|-----------|
| `contextvars.ContextVar` | Python 3.7+ per-context storage | Yes | Yes (works in asyncio tasks and threads) | Low |
| `threading.local` | Per-thread storage | Yes | No (same thread, different coroutines share) | Low |
| Run-local dict keyed by `run_id` | Passed as argument | Yes | Yes | Medium (API change) |

**Recommendation:** `contextvars.ContextVar` is the correct choice. The SSE path runs `_run_streaming` in `anyio.to_thread.run_sync`, which creates a new thread per request — `contextvars` propagates the context snapshot into that thread. `threading.local` would also work but is less idiomatic for async-first code.

**Pattern:**
```python
# At module level in graph.py
import contextvars
_active_callbacks_var: contextvars.ContextVar[list[Any]] = contextvars.ContextVar(
    "_active_callbacks", default=[]
)

# In run() and SSE _run_streaming():
_active_callbacks_var.set([_handler] if _handler else [])
# ...
config={"callbacks": _active_callbacks_var.get()}
```

Remove `self._active_callbacks` instance field entirely. The SSE route handler (`routes/run.py`) also needs updating — it currently writes `orchestrator._active_callbacks` directly.

### Wave 2 Fix 3: SQLiteCheckpointStore WAL + Persistent Connection

**Confirmed problem:** `checkpoint_store.py` defines `_connect()` as a method that calls `sqlite3.connect(self.db_path)` each time. Every `save()`, `load_latest()`, and `list_checkpoints()` call opens and closes a new connection. WAL mode is never set.

**Contrast:** `SQLiteRunStore` in `storage/sqlite.py` already demonstrates the correct pattern:
- Opens one persistent connection in `__init__`
- Sets `PRAGMA journal_mode=WAL` and `PRAGMA busy_timeout=5000`
- Uses a `threading.Lock()` to serialize access
- Stores `self._conn`

**Fix pattern:** Replicate `SQLiteRunStore`'s connection management in `SQLiteCheckpointStore`:
```python
def __init__(self, db_path: str = ".tmp/langgraph_checkpoints.db") -> None:
    self.db_path = Path(db_path)
    self.db_path.parent.mkdir(parents=True, exist_ok=True)
    self._lock = threading.Lock()
    self._conn = sqlite3.connect(str(self.db_path), check_same_thread=False)
    self._conn.row_factory = sqlite3.Row
    self._conn.execute("PRAGMA journal_mode=WAL")
    self._conn.execute("PRAGMA busy_timeout=5000")
    self._initialize_schema()

def _initialize_schema(self) -> None:
    with self._lock:
        self._conn.executescript(...)
        self._conn.commit()

def save(self, ...) -> None:
    with self._lock:
        self._conn.execute(...)
        self._conn.commit()
```

Remove `_connect()` method. Add `close()` method (matching `SQLiteRunStore`).

**Thread safety note:** `_finalize()` in `graph.py` calls `checkpoint_store.save()`. The SSE streaming path calls it from a thread via `anyio.to_thread.run_sync`. The `threading.Lock()` guard is required.

### Wave 2 Fix 4: `seen_tool_signatures` → Set

**Confirmed problem:** `state_schema.py:85` declares `seen_tool_signatures: Annotated[list[str], operator.add]`. Membership check at `graph.py:1683` uses `if signature in state["seen_tool_signatures"]` — O(n) list scan. The `operator.add` reducer runs on every graph edge merge, concatenating the full list.

**Migration impact:** Several files reference `seen_tool_signatures`:
- `state_schema.py`: declaration + `new_run_state()` + `ensure_state_defaults()`
- `graph.py`: `_ANNOTATED_LIST_FIELDS` frozenset + `_execute_action()` membership check + `.append()` call + `_dedup_then_tool_node()` membership check (line 310)
- `test_state_schema.py`: `test_seen_tool_signatures_has_annotated_reducer()` — this test MUST be updated (it asserts `Annotated` + `operator.add` which will no longer be true)
- `test_annotated_list_fields_synchronized()` — must still pass after `seen_tool_signatures` is removed from `_ANNOTATED_LIST_FIELDS`

**Fix pattern:**
```python
# state_schema.py — remove Annotated wrapper
seen_tool_signatures: set[str]  # plain set, no reducer

# new_run_state() — initialize as set
"seen_tool_signatures": set(),

# ensure_state_defaults() — initialize as set, convert list→set if loaded from checkpoint
if "seen_tool_signatures" not in state_dict:
    state_dict["seen_tool_signatures"] = set()
elif isinstance(state_dict["seen_tool_signatures"], list):
    state_dict["seen_tool_signatures"] = set(state_dict["seen_tool_signatures"])

# graph.py _execute_action() — .add() instead of .append()
state["seen_tool_signatures"].add(signature)

# graph.py _ANNOTATED_LIST_FIELDS — remove "seen_tool_signatures"
```

**Serialization hazard:** SQLite checkpoint stores serialize state via `json.dumps`. Python `set` is not JSON-serializable. Two approaches:
1. Serialize as list (in `json.dumps` default handler) and deserialize back to set in `ensure_state_defaults`. The `default=str` in checkpoint's `json.dumps` won't handle sets — need `default=lambda x: list(x) if isinstance(x, set) else str(x)`.
2. Store as `list` in state, convert to `set` only for lookups inside `_execute_action`. This avoids serialization issues but doesn't get the full benefit.

**Recommendation:** Option 1. Update `json.dumps` in `checkpoint_store.py` to serialize sets as lists. Update `ensure_state_defaults()` to convert list→set after deserialization. This is the cleanest approach with actual O(1) lookups.

### Wave 2 Fix 5: Cap `pipeline_trace`, `handoff_queue`, `handoff_results`

**Confirmed locations:**
- `pipeline_trace`: nested inside `state["policy_flags"]["pipeline_trace"]` — a list appended in `_emit_trace()` at `graph.py:475-477`. Not a top-level RunState field.
- `handoff_queue`: top-level `list[dict[str, Any]]` in RunState, no reducer, never evicted. Populated via `state["handoff_queue"].append(...)`.
- `handoff_results`: top-level `list[dict[str, Any]]` in RunState, no reducer, never evicted. Populated via `state["handoff_results"].append(...)`.

**Fix pattern for `_emit_trace()`:**
```python
def _emit_trace(self, state: RunState, stage: str, **fields: Any) -> None:
    policy_flags = state.get("policy_flags")
    if not isinstance(policy_flags, dict):
        return
    trace = policy_flags.setdefault("pipeline_trace", [])
    if isinstance(trace, list):
        trace.append({"stage": stage, "step": state.get("step", 0), **fields})
        # Oldest-first eviction
        if len(trace) > _PIPELINE_TRACE_CAP:
            del trace[: len(trace) - _PIPELINE_TRACE_CAP]
```

**Fix pattern for handoff lists:** Apply cap after each `.append()` call:
```python
state["handoff_queue"].append(...)
if len(state["handoff_queue"]) > _HANDOFF_QUEUE_CAP:
    state["handoff_queue"] = state["handoff_queue"][-_HANDOFF_QUEUE_CAP:]
```

**Recommended cap values (discretion):**
- `_PIPELINE_TRACE_CAP = 500` — enough for ~500 node transitions per run; typical runs are 20-100 steps
- `_HANDOFF_QUEUE_CAP = 50` — handoff queue is per-run; 50 pending handoffs is well above practical limits
- `_HANDOFF_RESULTS_CAP = 50` — same reasoning

**Whether to use env vars:** Hardcode as module-level constants. Env-var configurability adds complexity and these are internal guardrails, not user-facing limits. Note: `SQLiteRunStore` uses `MAX_RESULT_JSON_BYTES = int(os.environ.get(...))` as a pattern for env-var-gated caps — this is optional.

### Wave 3 Fix 6: Auto-Derive `_ANNOTATED_LIST_FIELDS`

**Confirmed location:** `graph.py:85-87`:
```python
_ANNOTATED_LIST_FIELDS: frozenset[str] = frozenset(
    {"tool_history", "memo_events", "seen_tool_signatures", "mission_reports"}
)
```

After the `seen_tool_signatures` set conversion (Fix 4), `_ANNOTATED_LIST_FIELDS` must be `{"tool_history", "memo_events", "mission_reports"}`. The auto-derivation replaces manual maintenance.

**Pattern (to replace the static frozenset):**
```python
import typing

def _derive_annotated_list_fields() -> frozenset[str]:
    hints = typing.get_type_hints(RunState, include_extras=True)
    result = set()
    for name, hint in hints.items():
        if typing.get_origin(hint) is typing.Annotated:
            args = typing.get_args(hint)
            if len(args) >= 2 and args[1] is operator.add:
                result.add(name)
    return frozenset(result)

_ANNOTATED_LIST_FIELDS: frozenset[str] = _derive_annotated_list_fields()
```

**Interaction with Fix 4:** After `seen_tool_signatures` is converted to a plain `set` (no `Annotated`), this derivation will automatically exclude it. The existing `test_annotated_list_fields_synchronized()` test validates this invariant and must continue passing.

**Note on execution timing:** `get_type_hints()` is called once at module import time (class-level statement). This is safe — `RunState` is defined before `_ANNOTATED_LIST_FIELDS` in the same module after the import is reorganized. Currently `_ANNOTATED_LIST_FIELDS` is in `graph.py` but `RunState` is in `state_schema.py`. The derivation should be added to `graph.py` (imports `RunState`).

### Wave 3 Fix 7: Consolidate State Initialization into `prepare_state()`

**Confirmed duplication:** `run()` (`graph.py:407-445`) and `_run_streaming()` (`routes/run.py:87-138`) each inline identical state initialization logic:
1. `new_run_state(system_prompt, user_input, run_id=run_id)`
2. Merge `prior_context` system messages into the system prompt
3. Merge prior conversation messages
4. `ensure_state_defaults(state, system_prompt=...)`
5. `state["rerun_context"] = ...`
6. `parse_missions(user_input)` + mission setup
7. `_build_mission_contracts_from_plan(...)` + `_initialize_mission_reports(...)`
8. Initial `checkpoint_store.save(...)`

**Extract pattern:**
```python
# In graph.py (method on LangGraphOrchestrator)
def prepare_state(
    self,
    user_input: str,
    run_id: str | None = None,
    *,
    prior_context: list[AgentMessage] | None = None,
    rerun_context: dict[str, Any] | None = None,
) -> RunState:
    """Single source of truth for run state initialization."""
    state = new_run_state(self.system_prompt, user_input, run_id=run_id)
    # ... prior_context merging ...
    state = ensure_state_defaults(state, system_prompt=self.system_prompt)
    state["rerun_context"] = dict(rerun_context or {})
    # ... mission parsing, contracts, reports ...
    self._write_shared_plan(state)
    self.checkpoint_store.save(run_id=state["run_id"], step=state["step"], node_name="init", state=state)
    return state
```

`run()` calls `self.prepare_state(...)`. The SSE route handler calls `orchestrator.prepare_state(...)` — this makes the private `_build_mission_contracts_from_plan` and `_initialize_mission_reports` calls go through the public method.

**Note:** The SSE route also calls `_write_shared_plan` and the initial `checkpoint_store.save` — both should move into `prepare_state()`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-run context isolation | Custom callback dict keyed by thread | `contextvars.ContextVar` | Built-in, async-safe, propagates into threads spawned via `anyio.to_thread.run_sync` |
| Serialize `set` to JSON | Custom encoder class | `lambda x: list(x) if isinstance(x, set) else str(x)` in `json.dumps(default=...)` | One-liner, already have `default=str` pattern in codebase |
| SQLite connection pooling | Custom pool | Persistent single connection + `threading.Lock()` | `SQLiteRunStore` already has this pattern — replicate it |
| Test parametrization over tool registry | Loop-based tests | `pytest.mark.parametrize` | Better error reporting, individual test identity |

## Common Pitfalls

### Pitfall 1: `seen_tool_signatures` Serialization Break

**What goes wrong:** After converting to `set`, `json.dumps(state)` raises `TypeError: Object of type set is not JSON serializable` in `checkpoint_store.save()`.

**Why it happens:** `SQLiteCheckpointStore.save()` calls `json.dumps(state, sort_keys=True, default=str)`. `default=str` converts unknown types to strings — `str({...})` produces `"{'sig1', 'sig2'}"` which is not valid JSON array syntax and cannot be round-tripped via `json.loads`.

**How to avoid:** Update `default=` to `lambda x: sorted(x) if isinstance(x, set) else str(x)` in `checkpoint_store.py`. In `ensure_state_defaults()`, add `elif isinstance(state_dict["seen_tool_signatures"], list): state_dict["seen_tool_signatures"] = set(state_dict["seen_tool_signatures"])` to handle deserialization.

**Warning signs:** Tests that round-trip state through checkpoint store will fail with `TypeError` or silently produce string-form set values.

### Pitfall 2: `test_seen_tool_signatures_has_annotated_reducer` Test Breakage

**What goes wrong:** After converting `seen_tool_signatures` to a plain `set`, the existing test `test_seen_tool_signatures_has_annotated_reducer` (test_state_schema.py:38-43) will fail — it asserts `Annotated` + `operator.add` metadata.

**How to avoid:** Update the test to assert `seen_tool_signatures` is a plain `set[str]` type hint (no `Annotated` wrapper). Also update `test_annotated_list_fields_synchronized` to ensure `seen_tool_signatures` is NOT in `_ANNOTATED_LIST_FIELDS`.

### Pitfall 3: `_active_callbacks` SSE Route Mutation

**What goes wrong:** After switching to `contextvars.ContextVar`, the SSE route handler in `routes/run.py:122-125` still mutates `orchestrator._active_callbacks` directly. If only `graph.py` is updated, the SSE path writes to the old instance field (which no longer exists or is no longer read).

**How to avoid:** Update both `graph.py` (run() method) and `routes/run.py` (`_run_streaming()` inner function) to call `_active_callbacks_var.set(...)`. Remove `self._active_callbacks` instance field from `__init__` and from the type annotation at line 162.

### Pitfall 4: `_ANNOTATED_LIST_FIELDS` Import Cycle

**What goes wrong:** Moving the auto-derivation function to `state_schema.py` might seem cleaner, but `graph.py` imports from `state_schema.py` — if `state_schema.py` imports from `graph.py` for `_ANNOTATED_LIST_FIELDS`, circular import.

**How to avoid:** Keep `_ANNOTATED_LIST_FIELDS` derivation in `graph.py` (which already imports `RunState` from `state_schema.py`). No import cycle.

### Pitfall 5: `prepare_state()` Breaking SSE Route Access to Private Methods

**What goes wrong:** The SSE `_run_streaming()` currently calls `orchestrator._build_mission_contracts_from_plan()` and `orchestrator._initialize_mission_reports()` directly. If `prepare_state()` is a public method on the orchestrator, it encapsulates these calls — but then the SSE route needs to call `orchestrator.prepare_state(...)` instead of accessing private methods. This is fine and is the goal.

**How to avoid:** The `prepare_state()` method should be a method on `LangGraphOrchestrator` (not a standalone function), since it calls `self._build_mission_contracts_from_plan`, `self._write_shared_plan`, `self.checkpoint_store.save`, etc.

### Pitfall 6: `run_bash` Disabled Breaks Existing Tests

**What goes wrong:** `test_run_bash_python3_passes`, `test_run_bash_python_in_path_not_blocked`, and `test_run_bash_empty_command` currently pass without `P1_BASH_ENABLED`. After adding the guard, they return a "disabled" error instead of executing.

**How to avoid:** Add `@pytest.fixture(autouse=True)` or `monkeypatch.setenv("P1_BASH_ENABLED", "true")` to the `run_bash` test file for tests that need execution. Keep a separate test for the disabled-by-default behavior.

### Pitfall 7: `pipeline_trace` Cap Must Preserve Diagnostic Value

**What goes wrong:** `pipeline_trace` is used for debugging pipeline stage progression. If the cap is too small, critical early stages are evicted before the run completes.

**How to avoid:** 500 is a safe cap — a typical multi-mission run with 10 missions × 20 steps × 2 trace events per step = 400 events. Cap at 500 preserves all events for typical runs while preventing unbounded growth in adversarial/runaway scenarios.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `threading.local` for per-call context | `contextvars.ContextVar` | Python 3.7+ | Works with asyncio tasks, anyio threads, concurrent.futures |
| New SQLite connection per query | Persistent connection + WAL mode | Long-established best practice | 10-100x reduction in connection overhead for write-heavy workloads |
| Manual frozenset maintenance | Auto-derivation from type hints at import time | Project-specific; follows "single source of truth" principle | Eliminates drift bugs when RunState fields are added/removed |
| O(n) list membership for dedup | O(1) set membership | Fundamental Python | Significant for long runs with 100+ tool calls |
